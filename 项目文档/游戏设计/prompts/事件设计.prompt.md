根据设计目标设计游戏事件或选择系统。**只产出设计方案文档，不产生代码。**

**工作流引擎约束：本 prompt 的每一步骤执行前必须先调用 `python 工作流/设计引擎.py check <step_id>`，完成后调用 `python 工作流/设计引擎.py done <step_id> <摘要>`。步骤 ID 定义在 `工作流/设计路由.json` 的 `事件设计` 工作流中。内部自检时必须调用 `python 工作流/设计引擎.py audit` 并将结果嵌入预览。**

**来源类型标记（继承自 Agent 核心原则第 5 条）：**
每个选项的效果/消耗必须标注来源类型：
| 标记 | 含义 |
|------|------|
| ✅ 基于 | 项目文档中已存在的机制/事件/数值 |
| 💡 推断 | 基于已有机制合理延伸 |
| ✨ 新机制 | 自主创造，文档中不存在 |

## 核心原则（正向引导）

1. **每个选项必须有实质性差异** — 去掉 log 文本后，选项仍可区分
2. **资源转换 > 数值比较** — 选项间应是不同资源类型的转换，而非"选最大的数"
3. **事件应有合理的触发条件** — 避免不合逻辑的触发
4. **选项应体现战略权衡** — 玩家选择时应感受到"我放弃了什么，得到了什么"

## 步骤

### 0. 读取目标游戏事件数据模型

在开始设计前，先确定本游戏中的"事件"包含哪些字段。

方法：
- 搜索项目中已有的事件定义（如 `ZHANGUO_EVENTS`、`QIN_EVENTS`）
- 分析游戏机制对事件的特殊需求（触发条件格式、选项效果类型、是否可以给事件加新选项等）
- 输出字段清单

### 1. 情景分析

**产出：玩家状态描述 + 玩家需求推断（一段话，必须展示）**

推断玩家此时所处的局面：
- 游戏阶段（初期/中期/后期？）
- 当前的核心矛盾（缺钱？缺人口？遭遇危机？）
- 玩家此刻可能的心态（希望平稳发展？期待变化？）

### 2. 设计目的确立

**产出：一句话设计目标（必须展示）**

明确回答：**这组事件要带给玩家什么样的选择体验？**

### 3. 概念发散

**产出：3-5 个候选场景/冲突类型，每个附一句话核心体验 + 参考来源（必须展示）**

围绕设计目的发散可能的事件主题/冲突类型。

```
引用来源：跨项目知识/选择与决策设计通用规则.md

概念A"天灾威胁"：核心体验是"不可抗力前的最优资源配置"
概念B"商路机遇"：核心体验是"短期利益vs长期投资的抉择"
概念C"内部矛盾"：核心体验是"人情与效率的两难"
```

### 4. 行为预检

按 agent.md 中的行为预检框架执行六维度分析。以下是事件设计的特有提问角度：

**选择情境补充**
- 事件触发时玩家正处在什么状态？（正常发展中 / 刚经历危机 / 资源紧张）
- 玩家是否有可能因为当前状态而倾向于某个选项？（如缺钱时倾向选金的选项）
- 是否有选项看起来是"陷阱"（看似好但实际弊大于利）？玩家能识别出来吗？

**内在驱动力补充**
- 每个选项是否提供了不同的"故事体验"而不只是数值差异？
- 选项之间是否有价值观/策略风格的冲突（如"短期生存 vs 长期投资"）？
- 玩家选择后是否能感受到"这是我的决定导致的后果"？

**外在驱动力补充**
- 各选项的收益是否直观可见？还是需要计算才能比较？
- 如果有隐藏收益（如获得卡牌），玩家能否感知到其价值？

**排斥力补充**
- 是否有选项要求玩家牺牲已经积累的资源？（触发损失厌恶）
- 选项的代价是否清晰？玩家能否准确判断"划不划算"？

**事件特有风险模式**
- 🔴 **永远最优**：将所有收益换算为同种资源后，某个选项全面领先 → 调整收益类型或量级
- 🔴 **两害相权**：所有选项都是负面效果 → 改为资源转换（牺牲资源A换取资源B）
- 🔴 **数值碾压**：某个选项的数值显著高于其他选项且消耗类型相同 → 错开资源类型
- 🔴 **信息不对称**：需要游戏机制深度知识才能做出最优选择 → 考虑加提示或简化

**产出：六维度驱动力分析 + 设计目的校对结论。** 通过预检后进入具体设计。

### 5. 具体设计

**产出：按步骤 0 确定的事件数据模型，输出完整事件表（必须展示）**

- **触发条件**：何时触发、附加条件
- **选项设计**：每个选项的资源转换关系
- **选项差异验证**：去掉文本后是否仍可区分
- **联动效应**（如有）：事件是否与卡牌/系统产生交互

```
引用来源：家族纪元/原型/engine/core.py / ZHANGUO_EVENTS

| 事件 | 触发 | 选项 | 效果 | 消耗 | 来源 | 设计意图 |
|------|------|------|------|------|------|----------|
| ... | ... | ... | ... | ... | ... | ... |
```

### 6. 内部自检（不展示给用户）

完成草稿后，逐条检查以下清单。**全部通过后才能输出预览。**

```
[ ] 情景分析已完成（有明确的玩家状态描述）
[ ] 设计目的已确立（有一句话设计目标）
[ ] 概念发散已完成（至少 3 个候选概念）
[ ] 行为预检已执行（六维度分析完整）
[ ] 每个选项有实质性差异（去掉文本后仍可区分）
[ ] 无"两害相权"式负反馈选项
[ ] 选项间是资源转换而非数值大小比较
[ ] 事件触发条件合理
[ ] 将所有选项收益换算为同一种资源时，无"永远最优"
[ ] 极端情况已考虑（资源不足时选项的行为）
```

如果发现违规 → **自动修正设计** → 重复自检，直到全部通过。

### 7. 输出设计方案预览（展示给用户）

通过自检后，按以下格式输出完整预览。**每步的推论和引用来源必须保留。**

```
## 事件设计方案

### 使用的引用
- 事件数据模型：家族纪元 ZHANGUO_EVENTS / QIN_EVENTS
- 设计原则：跨项目知识/选择与决策设计通用规则.md

### ① 情景分析
[玩家状态描述]

### ② 设计目的
[一句话设计目标]

### ③ 概念发散
[候选概念列表]

### ④ 行为预检
[六维度分析结论]

### ⑤ 具体设计

| 事件 | 触发 | 选项 | 效果 | 设计意图 | 来源 |
|------|------|------|------|----------|------|
| ... | ... | ... | ... | ... | ... |

#### 选项差异验证
- 选项A vs 选项B：涉及 [不同资源类型]，无永远最优
```

预览后等待用户确认。用户可要求修改，修改后重复 5→6。

### 8. 输出最终设计规格书（确认后）

用户确认后，输出一份完整的、可供实现者直接使用的设计规格书。

**文件位置规则：**
- 用户确认前：规格书保持在 `项目名/临时/<名称>.md`
- 用户确认后：**必须将文件从 `临时/` 移至 `项目名/设计文档/<名称>.md`**
- 退回则留在 `临时/`，修改后再次预览，确认后再移动
