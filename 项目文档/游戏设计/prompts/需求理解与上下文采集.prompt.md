将用户的设计需求拆解为明确的设计目标清单，同时搜索项目中已有的相关设计、知识作为参考。

## 步骤

### 1. 确认需求范围
- 用户最想解决的问题是什么？成功的标准是什么？
- 本次设计包含什么？不包含什么？
- 与现有系统的边界在哪里？

### 2. 文件发现——定位相关上下文

先了解项目结构，再按图索骥搜索具体文件。不要在未知结构的情况下盲目搜索。

**发现策略（强制执行）：**
1. **先读 `_索引.md`** — 项目根目录的索引文件包含全部文件列表和描述（由 `generate_index.py` 自动生成）
2. 从索引中找到与设计任务相关的目录和文件
3. 读取 `_项目配置.md`（如有），获取项目的配置和映射
4. 在定位到的目录中搜索具体文件

**产出要求：** 上下文摘要的"已有上下文"表中，**每行的"来源"必须标注"索引→文件路径"**，例如：

```
| 来源 | 位置 | 关键内容 |
|------|------|----------|
| 索引→朝代设计 | 朝代设计/三国_完整设计.md | 三国职业、势力站队机制 |
| 索引→设计 | 设计/_卡牌效果数据库.md | 各职业卡牌数值 |
```

缺少"索引→"前缀 → 自检不通过，退回重新搜索。

### 3. 同步搜索上下文

在设计需求确认的同时，搜索以下来源获取参考信息：

- **设计文档**：从索引确定的设计目录中读取设计文档、GDD
- **已有数据**：卡牌池、事件列表、职业定义、经济参数
- **跨项目知识**：`_共享资源/知识库/跨项目知识/` 中的通用设计规则
- **技能文件**：`_共享资源/技能/` 中的相关 skill

搜索时指定目标范围（如"三国相关"、"经济系统相关"），避免泛搜。

### 3. 整理摘要

输出合并后的结构化摘要，包含：

```
## 需求确认
- 目标：...
- 范围：包含 ...，不包含 ...
- 核心体验：...

## 已有上下文
| 来源 | 位置 | 关键内容 | 可用性 |
|------|------|----------|--------|
| 设计文档 | path/to/doc.md | 已有职业定义 | 部分可用 |
| 知识库 | path/to/rule.md | 通用设计原则 | 直接可用 |

## 经济参数（如需）
- 当前时代税率/产出倍率
- 人口上限/住房成本
- 已有职业的基准数值（作为新设计的参照系）
```

## 产出

一份包含需求确认和上下文采集的摘要报告。用户确认后进入方案设计阶段。
