# 快乐优先设计原则——先保证快乐总量，再压缩无效进程

> 来源：构筑密度框架设计实践——多轮自问自答后的经验教训总结

---

## 背景

在构筑密度框架的设计过程中，我们试图最大化"快乐密度"（快乐总量/进程）。最初的假设是：
- 进程越短 → 分母越小 → 密度越大
- 所以要把每战压缩到 40-60 秒

经过多轮自问自答后发现此路不通——进程过短时，快乐总量也被压缩。最终得出的正确方向是：

---

## 核心原则

### 先快乐，再优化

```
原思路（错误）：
  快乐密度 = 快乐总量 / 进程
  → 先定进程（越短越好），再往里塞快乐
  → 结果：快乐塞不进去，密度反而低

正确思路：
  快乐密度 = 快乐总量 / 进程
  → 先设计玩法，保证快乐总量足够大
  → 再在不破坏快乐总量的前提下，压缩无效时间
  → 快乐密度自然大
```

### 为什么先定进程是错的

进程过短时，根本产生不了真正有意义的快乐：

| 进程长度 | 能承载的快乐 | 评估 |
|:---------|:------------|:------|
| 40-60秒 | 简单选牌 + 摊牌结果 + 奖励 | ❌ 构筑感不成立，快乐总量低 |
| 2-5分钟 | 跨战积累 + 策略调整 + 效果验证 | ✅ 快乐总量足够 |
| 10分钟+ | 深度构筑 + 极限优化 | ✅ 但需要检查是否有无效时间 |

### 压缩策略——消除不做决策的时间，不缩减决策本身

先设计出有趣的玩法，然后再找出其中**不做决策的时间**来压缩：

```
可压缩的无效时间：
  · 过长的动画播放（可跳过/加速）
  · 等待对手响应（改为同时摊牌）
  · 无意义的操作步骤（自动处理）
  · 冗余的信息展示（简化为关键信息）

不可压缩的决策时间：
  · 选牌时的策略思考
  · 调整构筑方向的判断
  · 理解对手模式的分析
```
