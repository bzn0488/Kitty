# 三端分工设计

> 标签：`game-architecture`, `system-interaction`, `game-design`
> 关联概念：`18-快乐总量.md`, `34-系统交互与体验调节.md`

## 定义

将游戏系统的功能输出划分为三个独立端的设计模式：启动端（产能）→ 数值端（转化）→ 终端（输出）。三端各自独立运作，通过数据流而非直接调用耦合。

## 三端模型

```
启动端（产能）──→ 数值端（转化）──→ 终端（输出）
    │                      │
    │  金币/资源            │  攻/防/治疗
    │  经济单位             │  战斗单位
    │  任务系统             │  技能系统
    └───────────────────────┘
        间接影响（如任务→经济）
```

### 各端定义

| 端         | 功能                 | 典型系统                     | 设计要点                             |
| :--------- | :------------------- | :--------------------------- | :----------------------------------- |
| **启动端** | 产生资源/货币        | 经济单位、商店、任务系统     | 不能直接影响输出，只能提供资源供转化 |
| **数值端** | 将资源转化为战斗数值 | 卡牌属性、技能装配、升星     | 转化效率决定构筑强度                 |
| **终端**   | 最终输出到游戏世界   | 战斗系统、技能效果、自动攻击 | 只读取数值端的结果，不参与资源管理   |

## 设计要点

### 三端解耦的好处

1. **单端修改不影响其他端**：调整经济产出不需要修改战斗数值
2. **中间层涌现**：启动端和数值端通过共享中间层关联，而非直接调用
3. **线性可控**：每一端都是独立的"输入→处理→输出"环节，便于调试和平衡

### 三端耦合的风险

| 问题             | 示例                     | 后果                |
| :--------------- | :----------------------- | :------------------ |
| 启动端直通终端   | 经济单位直接产出伤害     | 平衡失控，经济=战斗 |
| 数值端依赖启动端 | 战斗数值直接由金币量决定 | 穷的越穷，富的越富  |
| 终端反向影响     | 战斗表现反作用于资源产出 | 循环难以预测        |

## 项目实践

参考增量自走棋的框架设计（`项目经验/06-增量自走棋-经济与战斗分路设计.md`）：

- **启动端**：经济单位产出金币、任务系统产代币
- **数值端**：商店购买卡牌→卡牌属性+技能装配→构筑强度
- **终端**：战斗系统自动执行、卡牌技能自动释放

三端通过"金币"和"卡牌"两个中间层关联，不直接耦合。
