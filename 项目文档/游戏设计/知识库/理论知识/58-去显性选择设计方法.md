# 去显性选择设计方法

> 来源：千水知乎《关于去显性选择》
> 关联：`理论知识/22-权力下放设计.md` `理论知识/41-涌现式设计.md` `理论知识/34-系统交互与体验调节.md`

---

## 一、什么是显性选择

**显性选择**：游戏直接给玩家列出选项，让玩家从中选取。

最常见的形式是"三选一"——杀戮尖塔中每场战斗后从三张卡牌中选择一张。

**显性选择的问题**：
- 玩家可能感受到"选择的压力"（不知道选哪个好）
- 选择过于符号化，缺乏与游戏世界的连接
- 玩家可能在无法评估选项价值时随意选择
- 当选项过多时，选择本身变成一种负担（选择的悖论）

---

## 二、去显性选择的价值

去显性选择不是"不给玩家选择"，而是**让选择从游戏行为中自然涌现，而非浮于界面之上**。

| 维度           | 显性选择                   | 隐性/去显性选择                |
| :------------- | :------------------------- | :----------------------------- |
| **表现方式**   | 界面弹窗、选项列表         | 玩家行为自然触发               |
| **认知负荷**   | 高——需要阅读和比较         | 低——玩家在做"自然的事"         |
| **沉浸感**     | 低——打断游戏流程           | 高——不脱离游戏世界             |
| **错误容忍度** | 低——选错了有明显的"损失感" | 高——玩家甚至不知道自己在选择   |
| **典型场景**   | 卡牌三选一、分支对话       | 移动路线选择、战斗中的策略判断 |

## 三、去显性选择的实现方法

### 3.1 行为即选择

玩家在做一件"自然"的事情时，实际上在做选择：

| 例子      | 表面行为       | 实际选择        |
| :-------- | :------------- | :-------------- |
| 旷野之息  | 朝某个方向走   | 选择了探索路线  |
| 黑魂      | 使用某种武器   | 选择了Build方向 |
| Minecraft | 去某个方向采集 | 选择了资源类型  |

### 3.2 环境引导

利用视觉/物理线索引导玩家，而非直接告诉玩家：

- 旷野之息的高塔 = "去那里，你能看到更远的地方"
- 光点/亮点 = "那里有东西值得探索"
- 地形起伏 = "这条路通向某个地方"

### 3.3 渐进式暴露

不让玩家一开始就看到所有选项，而是随着游戏进程自然解锁：

```
初期：只有基础操作可用（父爱逻辑）
中期：更多可能性解锁（母爱逻辑）
后期：玩家自创玩法（涌现式）
```

---

## 四、去显性选择的优劣分析

| 优势                     | 劣势                         |
| :----------------------- | :--------------------------- |
| 沉浸感更强               | 玩家可能不知道"自己在做选择" |
| 认知负荷更低             | 可能被理解成"缺乏选择"       |
| 更接近真实世界的行为模式 | 对设计者的要求更高           |
| 重玩性自然提升           | 不直观，需要玩家适应         |

---

## 五、设计自检

- [ ] 选择是否可以从界面中移到游戏行为中？
- [ ] 玩家是否能在不知情的情况下做出有效选择？
- [ ] 核心选择是否有足够的信息支持？
- [ ] 去显性化是否会降低玩家对选择的理解？
- [ ] 是否有混合方案（部分显性+部分隐性）？
