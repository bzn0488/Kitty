# 机制轴设计方法

> 标签：`game-mechanics`, `emergence`, `system-interaction`
> 关联概念：`41-涌现式设计.md`, `35-协同效应.md`, `04-叠层设计与延迟满足.md`

## 定义

设计游戏机制时，先定义成长依赖的"轴"（时间/击杀/资源/站位/血量等），让其他单位自然配合该轴，而非硬写条件触发。

## 为什么需要机制轴？

### 问题

设计时容易陷入"友方用技能→触发→获得"的套路，导致多张卡底层逻辑雷同——换了个名字但触发条件一模一样。

### 解法

设计单位时先问：**"它的成长依赖什么轴？"** 然后把"加速这个轴"的能力留给其他单位去自然配合。

## 常见机制轴

| 轴类型     | 依赖因素        | 例子                          | 自然协同方                   |
| :--------- | :-------------- | :---------------------------- | :--------------------------- |
| **时间轴** | 战斗时长        | 摇摇木马：随时间叠层→越久越强 | 充能单位（加速CD）、减CD装备 |
| **击杀轴** | 敌人死亡次数    | 发条战士：击杀→重置CD→再杀    | 压血线AOE、高攻速单位        |
| **资源轴** | 可消耗资源量    | 无限弹簧：消耗子弹→生成复制体 | 子弹补给单位、弹药上限提升   |
| **站位轴** | 格子位置关系    | 队长：攻击沿位置传递→递增     | 阵型排列、位置互换           |
| **血量轴** | 友方/自身生命值 | 医师：锁定低血量→双倍治疗     | 护盾提供者、减伤单位         |

### 机制轴 vs 条件触发

|  维度  |      ❌ 条件触发（硬写）      |              ✅ 机制轴（留接口）              |
| :----: | :--------------------------: | :------------------------------------------: |
|  写法  |   "其他单位为X→自身获得Y"    |   自身机制依赖某轴→其他单位自然能加速该轴    |
| 发现感 |       玩家按说明书配合       |          玩家发现"原来X可以加速Y！"          |
| 扩展性 |  每加一个新单位就要写新触发  |        新单位只要作用于同轴就自然协同        |
|  例子  | "友方用技能时，自身获得战意" | 摇摇木马随时间叠层→充能单位加速叠层→自然协同 |

## 设计要点

### 核心原则

设计完成后，把效果描述中的所有条件性文本删除，只留机制本身——**这个机制是否天然需要外部配合才能达到峰值？**

- 是 → ✅ 机制轴设计（涌现式）
- 否 → ❌ 自闭环或说明书式设计

### 自检问题

- ✅ **好设计**：玩家自己发现"把A和B放一起很搭"——涌现式
- ❌ **坏设计**：技能描述告诉玩家"把A和B放一起"——说明书式

## 项目实践

参考增量自走棋的单位系统设计规范（`项目经验/07-增量自走棋-升星机制设计.md`），设计时先确定单位的机制轴方向，再将"加速该轴"的能力分散到其他单位中，让协同自然涌现。
