# 游戏测试与迭代方法论

> 调研目标：提供卡牌Roguelike游戏的系统性测试方法、迭代流程、信号识别，
> 让设计团队知道"测什么、怎么看、怎么改"。

---

## 一、测试的四个层次

### 1.1 分层测试模型

```
层次1：烟雾测试（Smoke Test）— 做出来了吗？
  目标：确认游戏能跑通、不报错、UI 不崩
  执行者：开发者
  频次：每次改代码后

层次2：功能测试（Functional Test）— 对不对？
  目标：所有按钮能用、流程正确、数值计算无误
  执行者：开发者 + QA
  频次：每周

层次3：平衡测试（Balance Test）— 平衡吗？
  目标：各个构筑路线都能通关、没有必选卡、没有废卡
  执行者：有经验的玩家
  频次：每轮大更新后

层次4：体验测试（Experience Test）— 好玩吗？
  目标：玩家是否感到决策有深度、失败是否公平、是否有"再来一局"冲动
  执行者：目标用户群
  频次：里程碑节点
```

**核心原则**：在进入层次3之前，先确保层次1和2通过。在层次1有bug的游戏上测平衡是无意义的。

### 1.2 DBG卡牌游戏的测试特殊性

| 测试维度         | 传统游戏              | DBG卡牌Roguelike           | 测试难度 |
| ---------------- | --------------------- | -------------------------- | -------- |
| **测试变量数**   | 有限（角色/关卡固定） | 极大（构筑×遗物×地图随机） | 🔴 高     |
| **一次测试覆盖** | 能覆盖大部分路径      | 只能覆盖极小部分可能组合   | 🔴 高     |
| **平衡窗口**     | 可通过数值调整        | 构筑协同常产生意外突破点   | 🟡 中     |
| **体验评估**     | 看一段流程就知道      | 需要完整跑完一局           | 🟡 中     |
| **重现bug**      | 固定步骤可复现        | 随机组合下极难复现         | 🔴 高     |

## 二、测试前的准备

### 2.1 测试清单（出测前检查）

```
□ 烟雾测试通过（不报错、不崩溃）
□ 核心战斗循环完整（打牌→结算→怪物行动→下一回合）
□ 基础卡牌效果正确（所有关键字效果已验证）
□ 胜利/失败条件正确（通关、阵亡判定准确）
□ 商店/奖励系统可用（拿牌、删牌、买遗物都正常）
□ 至少准备3套预设构筑用于测试（不同路线各一套）
□ 有一个"快速通关"的开发者作弊手段（跳过/秒杀/直接胜利）
```

### 2.2 预设构筑的准备

不要每次测试都从零开始——准备几套预设构筑可以大幅提高测试效率：

```
构筑A：速攻型（低费堆叠路线）
  核心卡：[列表]
  关键协同：[描述]
  预期强度：[预期在第几幕成型、预期伤害/回合]
  测试目的：验证低费路线的上限

构筑B：控制/防御型
  核心卡：[列表]
  关键协同：[描述]
  预期强度：[预期生存能力、控制覆盖率]
  测试目的：验证控制路线的可行性

构筑C：高费爆发型
  核心卡：[列表]
  关键协同：[描述]
  预期强度：[是否有补偿机制、高费卡是否可用]
  测试目的：验证高费路线的可行性
```

## 三、测试过程中的观测要点

### 3.1 需要记录的五大类数据

```
战斗数据：
  □ 每场战斗的回合数（过长→构筑输出不足；过短→缺乏挑战）
  □ 每回合的操作数（操作太多→决策疲劳；太少→过于单调）
  □ 卡牌的打出率（收集→玩家拿了但从不打的卡是废卡）
  □ 费用的利用率（费剩太多→费用曲线有问题）
  □ 卡组大小变化（太胖→删牌不够；太瘦→构筑单一）

构筑数据：
  □ 各费用段的卡牌分布
  □ 各类型的卡牌分布（攻击/技能/能力比）
  □ 核心协同卡的获得时机（太早→碾压；太晚→没用）
  □ 删牌的频率和时机
  □ 升级的选择（哪些卡被优先升级）

玩家行为数据：
  □ 每步操作的时间（想太久→信息过载；秒选→无需思考）
  □ 三选一的选择率（某些卡永远不选→是废卡）
  □ Boss战的胜率（显著低于50%→难度问题）
  □ 中道崩殂的局原因分析（死于哪里？为什么？）

游戏进程数据：
  □ 每局总时长（过长→节奏慢；过短→内容不足）
  □ 各幕的通过率（哪一幕最难/最简单）
  □ 金钱/资源的获取 vs 消耗平衡

主观反馈（测试者记录）：
  □ 最让测试者感到"爽"的时刻
  □ 最让测试者感到"沮丧"的时刻
  □ 最有策略感的决策
  □ 最无聊/机械的操作
  □ 拿到某张卡时的第一反应
```

### 3.2 红黄绿灯信号系统

每个信号用红/黄/绿三类标记：

```
🔴 红灯（立即修复）：
  - 游戏崩溃/卡死
  - 某种构筑100%通关（或0%通关）
  - 有玩家"必拿"的卡
  - 有测试者拒绝再玩的卡（"有这张卡我就不打了"）

🟡 黄灯（需要关注）：
  - 某构筑路线通关率显著高于其他（>20%差异）
  - 超过5张卡在三选一中从未被选
  - 某个Boss战显著高于/低于平均胜率
  - 某费用段"没有可选的好卡"
  - 测试者出现"我知道我会死但不想继续了"

🟢 绿灯（一切正常）：
  - 各构筑路线都有玩家在玩
  - 三选二中玩家需要纠结（不是一眼选或全跳过）
  - 输的时候玩家说"刚才应该选另一张牌"（而非"游戏不公平"）
  - 测试者主动要求"再来一局"
```

### 3.3 构筑路线可行性的判断标准

```
一条构筑路线是否可行的量化标准：

通关率：≥5%（在全部测试局中，走这条路线的局通关的比例）
  低于5% → 路线太弱，需要加强
  高于30% → 路线太强，可能需要削弱

代表性：≥10%的测试者在没有提示的情况下自然选择了这条路线
  低于10% → 路线的Build信号不够明显

满意度：测试者走这条路线的"还想再玩一次"比例
  低于50% → 路线虽然能赢但不够有趣

三条都满足 → 路线健康
任何一条不满足 → 路线需要调整
```

## 四、平衡问题的诊断与修复

### 4.1 常见平衡问题速查表

| 症状             | 可能原因                 | 诊断方法                       | 修复方案                   |
| ---------------- | ------------------------ | ------------------------------ | -------------------------- |
| 某构筑通关率>50% | 核心卡太强               | 看这张卡在胜利局中的出现率     | 削弱核心卡或增加限制条件   |
| 某构筑无人使用   | Build信号不足            | 问测试者"你知道这条路可以走吗" | 增加引导性卡牌             |
| 某张卡永远不选   | 太弱或条件太苛刻         | 看三选一中选择率               | 加强或重做                 |
| 某张卡永远必选   | 通用强卡                 | 看所有构筑路线中都拿它         | 削弱通用性，改为路线特化   |
| 所有构筑都一样   | 缺乏差异化支持           | 各路线核心卡不明确             | 增加路线专属卡             |
| 前期总是死       | 初期卡太弱或费用曲线过高 | 看第一幕失败率                 | 加强初始卡或降低前几场难度 |
| 后期总是赢       | 成长曲线溢出             | 看后期构筑强度                 | 增加后期挑战或削弱成长上限 |
| 战斗时间过长     | 输出不足或怪物HP过高     | 看平均战斗回合数               | 依情况调整输出或HP         |
| 玩家无事可做     | 费用利用率低             | 看每回合平均操作数             | 增加低费卡或抽牌手段       |

### 4.2 平衡调整的优先级

```
P0（立刻修）：游戏不能正常玩
  崩溃、卡死、流程卡住
  某条路线完全无法通关
  某张卡导致游戏自闭环

P1（本周修）：明显影响体验
  某构筑胜率异常（>40% 或 <5%）
  有"必选"或"必不选"的卡
  测试者出现明显的沮丧情绪

P2（本轮迭代修）：需要优化
  某路线胜率偏高/偏低但仍在范围内
  部分卡牌出场率低
  节奏稍快/稍慢

P3（后几轮再修）：锦上添花
  数值微调（±1-2伤害）
  卡牌效果文案优化
  非核心功能优化
```

### 4.3 一次只改一个变量

平衡测试中最常见的错误：**一次改太多，不知道哪个改动起了作用**。

```
错误的改法：
  "这张卡太弱了" → 伤害+3、费用-1、多抽1张、获得关键词
  → 改完变超模，但不知道是哪个改动导致的

正确的改法：
  "这张卡太弱了"
  第一轮：伤害+3 → 测完再看
  如果还不够：费用-1 → 测完再看
  如果还不够：再加效果
```

## 五、迭代循环

### 5.1 推荐的迭代周期

```
独立开发者/小型团队：
  每日：自己跑1-2局，记录主观感受
  每周：请1-2个朋友测，收集反馈
  每双周：较大的平衡调整

中型团队：
  每日：开发者自测（层次1-2）
  每周：内部测试（层次2-3），5-10局
  每轮（2-3周）：外部测试（层次3-4），收集数据

注意：不要在一天内改完所有发现的问题。
       留一些问题到下一轮——一次改太多你无法知道哪个改对了。
```

### 5.2 一个迭代循环的完整流程

```
1. 确定本轮要测试的重点（不贪多）
   例："本次测试重点在第二幕的Boss难度和速攻构筑"
   → 测试者会重点看这两个方面

2. 跑测试，收集数据
   至少 5-10 局有效数据
   记录每局的关键数据（见 §3.1）

3. 分析数据，识别问题
   列出所有🔴信号（立即处理）
   列出主要🟡信号（本轮处理）
   列出次要🟡信号（留到下一轮）

4. 修复P0-P1问题
   一次只改一个变量（见 §4.3）
   改完确保层次1-2测试通过

5. 下一轮测试
   检查上一轮修改是否有效
   检查是否有新问题冒出
```

### 5.3 知道什么时候该停下来

```
何时停止迭代某个设计：
  □ 连续2轮测试没有出现新的🔴信号
  □ 所有构筑路线的通关率在10-25%之间
  □ 三选一中最少被选的卡也有>15%的选择率
  □ 测试者说"我觉得差不多了"而不是"还有问题"

何时应该大改（不是微调）：
  □ 某个系统有超过5个🔴信号
  □ 核心循环玩家觉得不好玩（不只是不平衡）
  □ 连续3轮测试后某个问题仍然存在（说明方向错了）
```

## 六、Roguelike游戏的特殊测试问题

### 6.1 随机性带来的测试覆盖问题

```
问题：一局Roguelike能体验到的内容不到全部内容的30%。
      你不可能通过"多玩几局"来覆盖所有组合。

解决方案：
  1. 定向测试：预设构筑+固定种子 → 测试特定路线
  2. 断点测试：从游戏中间开始（跳过前期），直接测试后期
  3. 模拟测试：写脚本模拟大量对局，收集统计数据
  4. 焦点测试：每次只测一个系统（比如这次只测卡牌协同）
```

### 6.2 "无聊"比"不平衡"更难发现

```
不平衡：容易发现——某构筑太强/太弱、某卡永远不选
无聊：难以发现——玩家不觉得有趣但说不清为什么

无聊的信号：
  - 测试者赢了但不想再玩（"赢了，但也就那样"）
  - 测试者每回合的思考时间越来越短（"反正都一样"）
  - 测试者开始注意其他事物（玩手机/聊天 > 看屏幕）

无聊的原因：
  - 每回合的选择都是"最优解"（没有纠结）
  - 构筑成型后就没有新决策了（"后面都一样"）
  - 失败不是因为决策错误而是因为"运气不好"
  - 卡牌之间的差异化不够（每张卡感觉差不多）
```

### 6.3 "失败"的体验设计

Roguelike 的失败是必然的（每局都会死）。关键是**失败的体验应该让人想重来，而不是想弃坑**。

```
好的失败：
  "我刚才不应该选那张牌的——如果我选了另一张可能就赢了"
  → 玩家觉得是自己的决策问题，学到教训后想再来
  
坏的失败：
  "我没抽到想要的牌/Boss直接秒了我"
  → 玩家觉得是运气问题，再来也没有意义

测试中需要关注：
  - 玩家失败时的第一句话是什么？
  - 玩家是立即开始新游戏还是停下来思考？
  - 玩家对失败原因的解释是"我的错"还是"游戏的问题"？
```

## 七、游戏测试的常见认知陷阱

| #   | 陷阱                   | 表现                                     | 解法                     |
| --- | ---------------------- | ---------------------------------------- | ------------------------ |
| 1   | **开发者盲区**         | 开发者太熟悉游戏，无法模拟新手体验       | 找没玩过的朋友测前30分钟 |
| 2   | **确认偏误**           | 只注意支持自己设计的数据                 | 强制记录"异常值"         |
| 3   | **幸存者偏差**         | 只分析通关的局，忽略失败的局             | 失败局的数据更有价值     |
| 4   | **过度解读少量数据**   | 1-2局就下结论                            | 至少5-10局再判断         |
| 5   | **太依赖好友反馈**     | 朋友不好意思说不好玩                     | 匿名问卷或找外人测试     |
| 6   | **修复了症状没修病因** | 某路线太强→削弱→玩家转向其他同样强的路线 | 查根本原因               |

## 八、测试工具与技巧

### 8.1 低成本测试工具

```
最低配置：
  □ 一个记录表格（Excel/Google Sheets）
  □ 屏幕录制工具（记录操作过程）
  □ 一个计时器（记录每局时长）

推荐配置：
  □ 数据埋点（自动记录卡牌出场率/胜率等）
  □ 种子系统（固定随机值用于复现问题）
  □ 测试模式（跳过剧情/直接到指定关卡）
  □ 热更新（不改代码就能调数值）
```

### 8.2 种子系统的测试价值

固定种子 = 固定随机序列 = 可复现的测试环境：

```
种子系统的用途：
  1. 复现bug：用同一种子跑，一定能看到同一个bug
  2. 对比测试：同一种子换不同构筑 → 对比平衡性
  3. 演示：用固定种子展示特定内容（避免了随机的不确定性）

设计建议：
  - 在开发期开放种子输入功能
  - 记录每个bug对应的种子号
  - 给测试者发固定种子的测试任务
```
