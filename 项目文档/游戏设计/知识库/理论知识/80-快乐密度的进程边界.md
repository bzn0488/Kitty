# 快乐密度的进程边界——快乐总量/进程 的最佳区间

> 来源：构筑密度框架设计实践中的自问自答推导
> 关联概念：`理论知识/18-快乐总量.md`、`理论知识/79-相对最优解与三层变化模型.md`

---

## 一、快乐密度的常见误解

快乐密度公式是：

```
快乐密度 = 快乐总量 / 进程
```

一个常见的直觉是：**进程越小，密度越大。** 即通过无限压缩每段游戏的时间来最大化密度。

但这个直觉有边界——**进程过短时，快乐总量也会同步缩小**，因为"没有足够的进程来产生有意义的快乐"。

---

## 二、进程边界

### 2.1 快乐密度的三个区间

```mermaid
flowchart LR
    A[进程过短<br>快乐总量受限] --> B[最佳区间<br>密度最大] --> C[进程过长<br>密度下降]
```

| 区间 | 特征 | 表现 |
|:-----|:------|:------|
| **低效区（进程过短）** | 快乐总量被时间压制——来不及产生构筑感/沉浸感/成就感 | 玩家觉得"太浅了" |
| **高效区（进程适中）** | 快乐总量足够高，进程可控——有足够的"养"的过程，但无低价值时间 | 密度达到最大值 |
| **低效区（进程过长）** | 快乐总量高但进程过长——大量等待/操作/执行时间中快乐密度被稀释 | 玩家觉得"太慢了" |

### 2.2 构筑类游戏的最佳进程区间

根据构筑密度框架的设计实践（卡包摊牌系统），推测的最佳区间：

| 进程长度 | 能承载的快乐 | 密度评估 |
|:---------|:------------|:---------|
| 40秒/战 | 简单选牌+摊牌结果+奖励 | ❌ 太短——构筑感不成立 |
| 2-5分钟/战 | 跨战积累+微调+效果验证+对手模式识别 | 🟢 适中 |
| 10分钟+/战 | 深度构筑+逐步成型+极限调整 | 🟡 过长——但传统构筑类游戏证明可接受 |

### 2.3 推论

```
快乐密度最大化的设计策略，不是"无限压缩进程"，
而是"找到让快乐总量饱和的最小进程长度"。
```

这要求设计者确定：**多少进程才能产生"真正有意义的快乐"？**
