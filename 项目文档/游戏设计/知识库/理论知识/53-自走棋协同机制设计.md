## 一、协同系统的核心功能

自走棋的协同（Synergy/Bond/Trait）是驱动构筑决策的核心引擎：

| 功能         | 说明                         |
| :----------- | :--------------------------- |
| **构筑导向** | 告诉玩家"该往什么方向凑"     |
| **强度阶梯** | 协同层数越多，强度越高       |
| **阵容标识** | 定义不同Build的身份          |
| **决策锚点** | 玩家围绕协同做购买和站位决策 |

## 二、协同触发模型

### 2.1 基础触发方式

自走棋中最常见的协同触发方式——**按数量触发**：

```
场上拥有 X 个同Tag单位 → 触发第 N 层协同效果
```

| 层数  | 需要数量  | 效果强度 | 设计意图           |
| :---: | :-------: | :------: | :----------------- |
|   1   |  2个单位  |    弱    | 前期可凑，引导方向 |
|   2   | 3-4个单位 |    中    | 中期主力           |
|   3   | 5-6个单位 |    强    | 后期成型           |
|   4   | 7-8个单位 |   极强   | 大后期/特殊条件    |

### 2.2 不同触发模型对比

| 模型         | 机制                       |    代表     | 设计特点                 |
| :----------- | :------------------------- | :---------: | :----------------------- |
| **线性模型** | 每多1个单位→强度线性提升   |   早期TFT   | 简单直接，但缺乏决策峰点 |
| **阶梯模型** | 达到特定数量→触发质变      | TFT当前版本 | 有明确的"里程碑"感       |
| **滑动模型** | 数量越多强度越高（无台阶） |  酒馆战棋   | 平滑但有边际递减         |
| **独有模型** | 特定单位组合→特殊协同      |  特殊英雄   | 创造独特的构筑体验       |

### 2.3 阶梯模型的设计优势

阶梯模型是目前自走棋的主流选择：

```
例："刺客"协同
2刺客 → 全体刺客+10%暴击（引导）
4刺客 → 全体刺客+30%暴击（成型）
6刺客 → 全体刺客+50%暴击（质变）
8刺客 → 全体刺客+75%暴击（终极）
```

**设计优势**：
- 每层有明确的"门槛感"和"获得感"
- 不同层数适合不同游戏阶段
- 触发质变的层数 = 构筑的目标

## 三、Tag体系 vs 职业体系

### 3.1 两种体系对比

| 维度           | Tag体系             | 职业体系                           |
| :------------- | :------------------ | :--------------------------------- |
| **定义方式**   | 单位固有Tag         | 单位有职业+种族双属性              |
| **代表**       | 增量自走棋、前期TFT | TFT（职业+起源）、炉石酒馆（种族） |
| **协同维度**   | 单一维度            | 双维度交叉                         |
| **构筑灵活性** | 高——可混合          | 中——需双维度匹配                   |
| **复杂度**     | 低                  | 中高                               |

### 3.2 单维度 vs 多维度

```
单维度（Tag）：
    单位 A [Tag: 战士] ←→ 单位 B [Tag: 战士]
    所有战士单位共享"战士"Tag

双维度（职业+种族）：
    单位 A [职业: 刺客, 种族: 暗影]
    单位 B [职业: 刺客, 种族: 虚空]
    → A和B共享"刺客"协同
    → A和B的种族不同，可能触发不同种族协同
```

**设计选择**：
- 单维度：上手简单，决策路径清晰，但深度有限
- 双维度：构筑空间大，但新玩家需要学习更多信息

### 3.3 混合体系

现代自走棋倾向于混合多种协同类型：

| 协同类型     | 效果               | 例子             |
| :----------- | :----------------- | :--------------- |
| **种族协同** | 同种族单位获得增益 | 龙族、暗影、虚空 |
| **职业协同** | 同职业单位获得增益 | 刺客、战士、法师 |
| **英雄协同** | 特定英雄组合触发   | 星级/专属协同    |
| **装备协同** | 装备特定组合触发   | 装备合成/联动    |
| **临时协同** | 回合/阶段限时加成  | 特殊事件         |

## 四、协同强度曲线设计

### 4.1 强度曲线的三个区间

```
强度
↑
│         ● 终极协同（极高门槛，极高回报）
│        /
│       ● 主力协同（正常门槛，高回报）
│      /
│     ● 基础协同（低门槛，低回报）
│    /
│   ● 引导协同（极低门槛，引导方向）
└────────────────────────────→ 单位数量
```

### 4.2 设计原则

| 原则             | 说明                           | 反例                     |
| :--------------- | :----------------------------- | :----------------------- |
| **低门槛低回报** | 2个单位就能触发，效果很弱      | 2个单位就有质变→过渡太强 |
| **高门槛高回报** | 需要大量投资，效果显著         | 8个单位但效果普通→没人凑 |
| **边际递减**     | 越往后每增加一个单位的收益越低 | 线性增长→最深层太强      |
| **有上限**       | 协同有最高层数限制             | 无上限→无限堆叠          |

### 4.3 溢出协同的处理

当玩家拥有超过协同触发要求的单位数量时：

| 处理方式     | 效果                           | 代表         |
| :----------- | :----------------------------- | :----------- |
| **不溢出**   | 达到最高层后不再触发           | 标准TFT      |
| **微量溢出** | 超过部分提供极小增益           | 部分变体     |
| **替换溢出** | 溢出单位可以换成其他类型的协同 | 灵活构筑鼓励 |

## 五、协同设计的常见问题

### 5.1 "说明书式"协同

当协同触发条件过于机械（如"集齐XX种Tag"），玩家的体验是：
> "我只要找这些Tag就行" —— 而不是 "我发现了巧妙的组合"

**对策**：引用 `理论知识/35-协同效应.md` 中"涌现式协同"的设计原则——协同应基于共享中间层（行为/属性/状态），而非指名标签联动。

### 5.2 单一最优解

当某套协同明显强于其他所有方案时：

| 症状                | 原因           | 对策                   |
| :------------------ | :------------- | :--------------------- |
| 所有人追同一套Build | 某协同数值过高 | 平衡调整               |
| 不追这套就赢不了    | 协同差异化不足 | 增加协同之间的克制关系 |
| 赛季中期固化        | 内容消耗殆尽   | 赛季轮换+加入新单位    |

### 5.3 协同强度不足

当协同效果太弱，玩家"不在乎触不触发"：

| 症状                    | 原因           | 对策                   |
| :---------------------- | :------------- | :--------------------- |
| 纯堆高质量单位 > 凑协同 | 协同数值过低   | 提高协同收益           |
| 玩家不关注协同标签      | 协同效果不明显 | 强化视觉/数值反馈      |
| 2星单位 > 协同加成      | 单卡强度超标   | 平衡单卡 vs 协同的权重 |

## 六、设计自检清单

- [ ] 协同触发是否有明确的"引导→成型→质变"阶梯？
- [ ] 是否存在一种明显优于其他的协同（最优解）？
- [ ] 玩家是否可以在多个协同方向之间灵活转换？
- [ ] 协同效果是否有足够反馈（视觉/数值）？
- [ ] 协同设计是否符合"涌现式"而非"说明书式"？
- [ ] 是否有混合协同（不同协同维度交叉）的策略空间？
