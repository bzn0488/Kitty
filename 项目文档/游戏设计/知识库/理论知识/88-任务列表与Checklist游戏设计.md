## 一、定义与分类

### 1.1 什么是Checklist游戏设计

**Checklist**（任务列表/清单）在游戏设计中泛指一类**结构化的进度追踪机制**——游戏将可完成的目标以清单形式呈现给玩家，玩家通过逐项完成获得正反馈和进度感。

它不是一个独立的游戏品类，而是一种**横跨多个品类的设计模式**。

### 1.2 Checklist的四种存在形态

| 形态                 | 描述                                      | 典型示例                                   |
| :------------------- | :---------------------------------------- | :----------------------------------------- |
| **显性任务系统**     | 游戏内建的任务/使命日志，明确列出待办事项 | 《巫师3》任务日志、《魔兽世界》任务追踪    |
| **Checklist UI**     | 专门的清单界面，标记已完成/未完成条目     | 《蜘蛛侠》犯罪清单、《孤岛惊魂》兴趣点标记 |
| **进度条式隐式清单** | 用进度数值/百分比暗示未完成内容           | 成就系统进度、好感度进度条                 |
| **玩家自建清单**     | 玩家因游戏内容丰富而自行记录              | 《艾尔登法环》玩家自建Boss清单             |

### 1.3 为什么研究这个主题

Checklist是游戏中最普遍但最容易被忽视的设计工具。它既是驱动力来源，也是体验破坏者。本文档的目标是**完成对它的诊断**——理解它的运作方式、它的有效边界、以及它为什么无法通过优化解决自身的问题。

## 二、Checklist的心理学基础

### 2.1 清单效应——为什么打勾让人快乐

| 机制             | 原理                               | 在Checklist中的体现                      |
| :--------------- | :--------------------------------- | :--------------------------------------- |
| **蔡格尼克效应** | 人对未完成任务的记忆强于已完成任务 | 未勾选的条目产生心理张力，驱动玩家去完成 |
| **目标梯度效应** | 越接近目标，追求力度越强           | 进度条越接近100%，玩家越急迫想完成       |
| **固定比率强化** | 每N次行为获得奖励                  | 每完成N个任务解锁下一个奖励节点          |
| **损失厌恶逆用** | 看到"未完成"产生不适感             | 列表中未完成条目成为"认知负担"，驱动清除 |
| **自我效能感**   | 完成目标证明"我有能力"             | 每勾掉一项都巩固"我能行"的认知           |

### 2.2 进度条效应

> **玩家对进度条的敏感度超过对实际奖励的敏感度。**

- 0% → 动力低
- 10%-50% → 稳定上升
- 70%-90% → 动力陡增（目标梯度效应峰值）
- 99% → 强迫性完成

进度条本身是一种正反馈信号——即使没有实质性奖励，"看到进度增长"就足以维持行为。**但这也意味着进度条可以成为操控玩家的工具。**

### 2.3 完成度的认知负担

当玩家面前有太多未完成任务时，产生两种极端反应：

| 类型           | 反应      | 表现                                           |
| :------------- | :-------- | :--------------------------------------------- |
| **完成主义者** | 焦虑驱动  | 不完成所有任务不推进主线，可能导致疲劳弃坑     |
| **自由主义者** | 忽视/反抗 | 完全无视任务列表，按自己节奏玩，但可能错过内容 |

## 三、Checklist的任务类型学

### 3.1 按行为分类

| 类型                       | 核心行为          | 风险                 |
| :------------------------- | :---------------- | :------------------- |
| **杀怪型**（Kill）         | 击杀N个目标       | 沦为刷怪，策略密度低 |
| **收集型**（Gather/Fetch） | 收集N个物品       | 跑腿，同质化高       |
| **护送型**（Escort）       | 保护NPC抵达目标   | AI拉胯时极度负反馈   |
| **递送型**（Delivery）     | 将物品从A运到B    | 路径无趣时空洞       |
| **探索型**（Explore）      | 到达/发现指定位置 | 缺乏引导时迷茫       |
| **解谜型**（Puzzle）       | 解决特定谜题      | 难度不匹配时卡关     |
| **建造型**（Craft/Build）  | 合成/建造指定物品 | 材料收集过程枯燥     |
| **复合型**（Hybrid）       | 上述多种组合      | 复杂度高但体验最好   |

### 3.2 按清单结构分类

| 结构         | 特征                   | 例子                           |
| :----------- | :--------------------- | :----------------------------- |
| **线性链**   | A→B→C→D，不可跳步      | 传统RPG主线任务                |

| **并行清单** | 大量独立任务同时开放   | 《魔兽世界》地图任务群         |

| **动态清单** | 根据玩家行为增删条目   | 《中土世界：暗影魔多》复仇系统 |

## 四、Checklist的核心设计模式

### 4.1 节奏控制：3-7条黄金法则

屏幕上可见的未完成任务应控制在 **3-7 项**——少于3项让玩家觉得没事做，多于7项造成认知过载。

| 阶段       | 清单特征         | 设计要点                       |
| :--------- | :--------------- | :----------------------------- |
| **导入期** | 少量、高引导性   | 建立"我是谁、要做什么"的认知   |
| **扩张期** | 逐步增加平行任务 | 给玩家选择权                   |
| **高潮期** | 主线缩短，紧迫感 | 减少支线干扰                   |
| **收尾期** | 清理剩余任务窗口 | 提供"最终战前可自由探索"的预警 |

### 4.2 奖励结构

| 奖励类型                          | 即时性 | 强度  |
| :-------------------------------- | :----- | :---- |
| **即时反馈**（经验/金币/道具）    | 立即   | 低-中 |
| **累积反馈**（成就/章节解锁）     | 延迟   | 中-高 |
| **叙事反馈**（剧情推进/关系变化） | 延迟   | 高    |
| **探索反馈**（新区域/隐藏内容）   | 延迟   | 高    |

### 4.3 承诺机制

当玩家接受一个任务时，实际上是在向游戏做出一个隐式承诺——产生了"我必须完成它"的心理负担。允许玩家放弃/删除任务是缓解这一负担的最低要求。

### 4.4 10选1困境

> **当清单中有N个条目时，玩家往往只会去做驱动力最高的1个，其余N-1个成为"清单噪声"。**

三种根因：

| 根因               | 机制                                                   |
| :----------------- | :----------------------------------------------------- |
| **空间距离衰减**   | 任务目标越远，时间/精力成本越高，驱动力越低            |
| **吸引力梯度不足** | 玩家只看任务本身的趣味性，不会因"在清单上"就产生驱动力 |
| **叙事身份失调**   | 玩家角色已成长，但清单上还有低级任务（通马桶问题）     |

**10选1的连锁后果**：
```
N个任务 → 玩家只做1个 → 其余N-1个累积
    → 清单越来越长 → 认知负担加重
    → 玩家连清单都不想打开
    → 所有任务失去驱动力 → 弃坑
```

## 五、Checklist的结构性问题（核心论点）

**以下三个问题是结构性的——在Checklist框架内无法通过优化解决。**

### 5.1 蚂蚁区隔（Ant Segmentation）

无论怎么优化排序/推荐/分组，清单中的条目天然存在驱动力梯度。玩家永远只做驱动力最高的那1-2个。

**关键推论**：筛除当前最低驱动力条目后，新的最低驱动力条目又会出现。递归进行下去，最终条目数为零。**这不是设计执行的缺陷，是结构的必然。**

### 5.2 外在驱动力污染

Checklist 的结构本身就在说"这里有N件事等你去做"——这个结构天然产生外在驱动力。即使把界面设计得再温和，它也是一份待办清单，而待办清单的本质是"债务"。（参见 `09-过度理由效应.md`）

**规定式清单**（明确告诉你"必须做的"）会压过内驱。
**记录式清单**（只记录不催促）看似无害，但只要你打开清单看到"未完成"，蔡格尼克效应就开始工作——这仍然是在借用外在驱动力。

### 5.3 内容驱动的不可再生性

Checklist 的每一条目代表一份固定内容。玩家消费后，该条目被"勾掉"——它提供的内在驱动力永久消失了。而系统交互（见 `89-网状系统游戏设计.md`）可以产生可再生的体验。

## 六、Checklist范式的过时性

### 6.1 Checklist适合什么

Checklist 范式并非一无是处。它适合：

- **线性叙事游戏**：有明确的A→B→C剧情需要玩家按顺序体验
- **教程/新手期**：帮助玩家建立"这个游戏有什么"的认知地图
- **成就系统**：作为"事后清单"记录已完成的挑战（而非驱动行为）

### 6.2 Checklist不适合什么

Checklist 范式不适合追求**内在驱动力优先**的游戏设计，因为：

- 它的结构天然制造外在驱动力
- 10选1困境导致大量内容被浪费
- 内容消耗不可逆
- 清单疲劳驱逐玩家的探索欲

### 6.3 超越Checklist

Checklist 应当被视作**过渡性工具**而非**游戏内容的组织方式**。如果游戏的核心目标是让玩家自主探索、自发生成目标，Checklist 会成为障碍。

替代方案见 `89-网状系统游戏设计——后Checklist时代的游戏组织范式.md`，其中提出了：

- **种子机制**：给玩家"枪"而非"任务"
- **物品即目标**：背包里的物品在相关地点给予揭示感
- **系统交互**：玩A系统触发B系统的有趣变化
- **世界状态条**：隐性的、非显式的进度感知
- **驱动力放大器**：Easy Fun 放大 Hard Fun 的驱动力，而非解锁它

## 二、心理学基础

### 2.1 清单效应——为什么打勾让人快乐

**完成任务的快感来自多个心理机制的叠加：**

| 机制             | 原理                               | 在任务列表中的体现                         |
| :--------------- | :--------------------------------- | :----------------------------------------- |
| **蔡格尼克效应** | 人对未完成任务的记忆强于已完成任务 | 未勾选的条目产生心理张力，驱动玩家去完成   |
| **目标梯度效应** | 越接近目标，追求力度越强           | 进度条越接近100%，玩家越急迫想完成         |
| **固定比率强化** | 每N次行为获得奖励                  | 每完成N个任务解锁下一个奖励节点            |
| **损失厌恶逆用** | 看到"未完成"产生不适感             | 列表中未完成条目成为"认知负担"，驱动清除   |
| **掌控感诉求**   | 人类天然渴望有序和掌控             | 将混沌的游戏世界转化为可追踪、可掌握的清单 |
| **自我效能感**   | 完成目标证明"我有能力"             | 每勾掉一项都巩固"我能行"的认知             |

### 2.2 进度条的神奇力量——"虚拟完成度"的心理

进度条是"隐式清单"。实验表明：

> **玩家对进度条的敏感度超过对实际奖励的敏感度。**

- 当进度条显示 0% → 玩家动力低（"才刚开始"）
- 进度条显示 10%-50% → 动力稳定上升
- 进度条显示 70%-90% → **动力陡增**（目标梯度效应峰值）
- 进度条显示 99% → **强迫性完成**（差最后一个就不舒服）

**设计启示**：进度条本身是一种正反馈信号——即使没有实质性奖励，"看到进度增长"就足以维持行为。

### 2.3 完成度的认知负担

当一个玩家面前有太多未完成任务时，会产生两种极端反应：

| 类型           | 反应      | 表现                                             |
| :------------- | :-------- | :----------------------------------------------- |
| **完成主义者** | 焦虑驱动  | 不完成所有任务不推进主线，可能导致疲劳弃坑       |
| **自由主义者** | 忽视/反抗 | 完全无视任务列表，按自己的节奏玩，但可能错过内容 |

**折中方案**：允许玩家「隐藏」或「归档」不感兴趣的任务，减少视觉负担。

## 四、核心设计模式

### 4.1 清单的节奏控制

任务列表不应是"越多越好"。**清单密度需要节奏控制：**

| 阶段       | 清单特征             | 设计要点                             |
| :--------- | :------------------- | :----------------------------------- |
| **导入期** | 少量、高引导性任务   | 帮助玩家建立"我是谁、要做什么"的认知 |
| **扩张期** | 逐步增加平行任务     | 给玩家选择权，避免线性疲惫           |
| **高潮期** | 主线任务缩短，紧迫感 | 临近结局时减少支线干扰               |
| **收尾期** | 清理剩余任务的窗口   | 提供"最终战前可自由探索"的预警       |

**黄金法则**：屏幕上可见的未完成任务数量控制在 **3-7 项**——少于3项让玩家觉得没事做，多于7项造成认知过载。

### 4.2 主线 vs 支线的设计博弈

| 维度         | 主线任务             | 支线任务                  |
| :----------- | :------------------- | :------------------------ |
| **紧迫性**   | 高（关乎核心剧情）   | 低（可随时暂停）          |
| **奖励密度** | 中（以叙事奖励为主） | 高（以数值/装备奖励为主） |
| **耗时**     | 可预测（线性流程）   | 不可预测（探索自由度高）  |
| **失败惩罚** | 剧情推进受阻         | 无（可选，随时可重试）    |

**设计原则**：
- 主线提供**叙事动力**（"接下来会发生什么？"）
- 支线提供**探索动力**（"那边有什么好玩的？"）
- 支线不应成为"必须做完才能推进主线"的障碍（除非有特殊设计意图）

### 4.3 清单的奖励结构

清单类游戏最核心的问题：**每个条目被打勾时的回报是否足够？**

| 奖励类型                              | 即时性 | 强度  | 适用场景               |
| :------------------------------------ | :----- | :---- | :--------------------- |
| **即时反馈**（经验/金币/道具）        | 立即   | 低-中 | 所有简单任务的基础回报 |
| **累积反馈**（成就/段位/章节解锁）    | 延迟   | 中-高 | 多任务完成后的质变节点 |
| **叙事反馈**（剧情推进/角色关系变化） | 延迟   | 高    | 核心主线和高质量支线   |
| **探索反馈**（新区域/隐藏内容可见）   | 延迟   | 高    | 地图解锁/新功能揭示    |

**结合分段奖励设计**（参考 `07-分段奖励设计.md`）：
- 一个长任务应拆分为多段奖励，而非一次性在最终交付时发放
- 每完成一个小步骤都应有微小的正反馈（经验值/文字提示/音效）

### 4.4 清单作为"承诺机制"

当玩家接受一个任务时，实际上是在向游戏做出一个**隐式承诺**：

```
接受任务 → 进入"待办"状态 → 心理负担产生 → 驱动力形成
    ↑                                            │
    └──────── 清除负担（完成/放弃）───────────────┘
```

**设计要点**：
- 允许玩家放弃/删除任务（否则未完成任务成为永久负担）
- 放弃任务的代价应合理（不惩罚探索行为）
- 被放弃的任务可以"重新发现"而非永久失去

### 4.5 去目标化与清单设计的平衡

结合 `60-去目标化指引设计.md` 的批判——清单设计不是"要不要有"的问题，而是**清单的呈现方式是否尊重玩家自主感**的问题：

| 有害的清单设计            | 健康的清单设计                    |
| :------------------------ | :-------------------------------- |
| 强迫玩家按固定顺序执行    | 允许玩家自选顺序                  |
| 清单是唯一的引导方式      | 清单辅助其他引导（视觉/环境引力） |
| 未完成任务持续弹窗提醒    | 清单是静默的、可折叠的            |
| 提示"你还有N个任务未完成" | 提示"你已完成N个任务"             |
| 只有一种完成方式          | 允许多样化的完成路径              |

### 4.6 清单的核心困境："10选1"问题

这是显性规定式清单最普遍但最被忽视的困境——

> **当清单中有 N 个条目时，玩家往往只会去做自己最感兴趣的那 1 个，剩下的 N-1 个被自动忽略，成为"清单噪声"。**

这不是玩家的懒惰，而是行为选择的自然结果。驱动力模型（`21-驱动力模型.md`）可以解释：

```
玩家面对10个任务 → 对每个任务计算驱动力总值
                → 选出驱动力最高的1个去执行
                → 其余9个停留在清单中 → 成为视觉负担
                → 下次再选时 → 还是选那个最感兴趣的 → 循环
```

#### 4.6.1 "10选1"的三种根因

| 根因                 | 机制                                                                     | 典型场景                                                                |
| :------------------- | :----------------------------------------------------------------------- | :---------------------------------------------------------------------- |
| **① 空间距离衰减**   | 任务目标距离玩家当前位置越远，完成任务的时间/精力成本越高，驱动力越低    | 玩家在A区域，只做A区域的3个任务，B/C/D区域各7个任务永远不碰             |
| **② 吸引力梯度不足** | 玩家只看该任务本身的趣味性/奖励，不会因为"它还在清单上"就产生驱动力      | 10个任务中只有1个涉及玩家喜欢的玩法（如探索新区域），其他9个是跑腿/杀怪 |
| **③ 叙事身份失调**   | 玩家当前的叙事身份（勇者/骑士/领袖）与任务的内容不匹配时，产生"通马桶"感 | 剧情已成英雄，清单上还有"帮老奶奶找猫"——玩家心理上不愿意做              |

**叙事身份失调**是最隐蔽也最致命的——它不是任务设计有问题，而是**任务出现在错误的时间点**。玩家不会因为"清单上还有"就去清它，反而会因为"这种低级的任务还在"而对任务系统产生反感。

#### 4.6.2 "10选1"的连锁后果

```
10个任务 → 玩家只做1个 → 剩下9个不断累积
    → 清单越来越长 → 认知负担加重
    → 玩家连清单都不想打开了 → "眼不见为净"
    → 所有任务（包括那个想做的）都失去驱动力
    → 玩家失去方向感 → 弃坑
```

这是一个**正反馈恶性循环**——清单越长，玩家越不想看清单；越不想看清单，清单上的任务越失去意义。

#### 4.6.3 应对"10选1"的设计策略

| 策略               | 做法                                                        | 解决的问题      | 注意事项                                                             |
| :----------------- | :---------------------------------------------------------- | :-------------- | :------------------------------------------------------------------- |
| **空间聚类**       | 只显示玩家当前位置周边可做的任务，远方任务折叠/延迟显示     | ①空间距离衰减   | 避免玩家感觉"被限制在区域内"                                         |
| **动态优先级排序** | 根据距离/玩家最近行为/任务紧迫度自动排序，最相关的排最前    | ①②              | 需要信任系统的排序逻辑，否则玩家会手动翻找                           |
| **叙事阶段化修剪** | 当玩家叙事身份提升到某个阶段后，自动归档/完成低优先级旧任务 | ③叙事身份失调   | **必须**给玩家"任务被清理的仪式感"，而非默默消失                     |
| **任务衰减机制**   | 未被领取的清单条目在一段时间后自然消失/被新任务替换         | ②吸引力梯度不足 | 让玩家知道"错过了就不再回来"（但需平衡错过焦虑）                     |
| **清单容量上限**   | 严格限制同时可见的未完成任务数量（建议6-8条）               | 所有            | 最直接有效的手段，但需要游戏设计上配合——不把"接任务"作为内容解锁手段 |
| **允许主动清理**   | 玩家可以标记"不感兴趣"来移除条目                            | 所有            | 移除的条目应有机会重新触发（如再次遇到NPC），而非永久丢失            |

#### 4.6.4 "10选1"的进阶思考：每个条目的有效驱动力范围

假设一个任务列表达到了"健康"状态（6条，空间分布合理），玩家的实际选择行为依然遵循**驱动力排序**：

```
玩家看到6个任务：
  任务A：驱动力 90  ← 最相关，做这个
  任务B：驱动力 70  ← 可能做
  任务C：驱动力 50  ← 边缘
  任务D：驱动力 30  ← 大概率忽略
  任务E：驱动力 20  ← 忽略
  任务F：驱动力 10  ← 忽略
```

**关键洞察**：不是把条目从10降到6就解决了问题。问题是**每条任务的驱动力是否高到足以进入玩家的"考虑区间"**。

**设计策略**：
- **宁可砍掉无用条目**：如果一条任务在大部分玩家的大部分游戏时间里驱动力都很低，就不应该出现在清单里。把它换成更容易触发的条件事件（玩家路过时自动触发，而非在清单上挂着）。
- **提升低驱动力条目的"瞬时吸引力"**：当一个条目在清单上挂了太久，可以给它加一个"临时增益"（额外奖励/限时完成奖励/附近区域的关联提示），人为拉高它的驱动力。
- **不要给玩家"选"的机会**：如果清单上有10个任务但玩家实际上只适合做其中2个，那就不该显示10个。游戏应该通过环境线索/空间引导来暗示"现在适合做什么"，而非让玩家自己在列表中筛选。

### 4.7 进阶概念：任务列表的几个前沿设计方向

#### 4.7.1 反清单——让任务"不告而别"

大多数游戏的问题是**任务一旦被接受就永远赖在清单上**。一个有意思的设计方向是：**任务可以在不被完成的情况下，以有意义的方式消失**。

| 消失方式               | 机制                                         | 玩家感知                       |
| :--------------------- | :------------------------------------------- | :----------------------------- |
| **时间推移自然失效**   | 任务有"有效期"，过期后任务提供方找别人解决了 | "噢，来晚了"——世界真实感       |
| **与其他任务互斥**     | 选择A任务导致B任务自动关闭                   | "这个选择有代价"——决策重量     |
| **被更高的身份取代**   | 当玩家地位提升，低级任务被上级自动接管       | "我的人会处理"——成长仪式感     |
| **被世界状态改变覆盖** | 某个关键事件后，相关任务不再有意义           | 世界是动态的，不是静止的任务架 |

**核心原则**：任务消失时应给玩家一个**叙事理由**和**仪式感**，而非默默从列表中删除。

#### 4.7.2 任务列表的"叙事身份过滤"

这是解决**叙事身份失调**（通马桶问题）的关键方法——**任务系统应感知玩家的当前叙事身份，并据此自动过滤掉不匹配的条目**。

```
玩家身份 = 新兵 → 可接：送信/巡逻/训练
玩家身份 = 骑士 → 可接：剿匪/护送/领地防务（送信等低级任务自动关闭）
玩家身份 = 领主 → 可接：外交/战争/建设（不再出现在第一线）
```

**实现难点**：
- 需要游戏有明确的身份阶段划分（不是所有游戏都有）
- 身份回退时的任务处理（玩家做了某个选择导致降级）
- 有些玩家可能想保留低级任务作为"放松"（应允许手动锁定）

#### 4.7.3 "引力场"取代任务列表

受 `60-去目标化指引设计.md` 启发——如果任务列表的问题太多了，有没有可能**不用任务列表，同样实现目标引导**？

引力场模型：每个可交互的目标对玩家产生一个虚拟"引力"，引力强度取决于：
- **空间距离**（越近引力越强）——视觉上可见的目标比清单上文字的目标更有吸引力
- **玩家历史偏好**（经常做的事引力增强）——系统学习玩家的行为模式
- **叙事紧迫度**（重要事件引力飙升）——主线相关目标自动获得高优先级
- **新奇度**（未见过的事物引力短暂爆发）——探索新区域的瞬间驱动力

**当多个引力叠加时，玩家自然会被"最亮的那颗星"吸引过去，不需要任何清单界面。**

#### 4.7.4 "事后清单"——让清单成为记录的副产物

一个反直觉的设计思路：**不提前告诉玩家有哪些事可做，而是等玩家做了之后，再生成一个"清单"来记录他做过的事**。

```
传统方式：清单 → 玩家选择 → 执行 → 打勾
事后清单：玩家探索 → 做了某件事 → 系统记录 → "哦，原来这算一个目标"
```

**优势**：
- 完全消除清单焦虑——"清单"是"我已取得的成就"而非"我还欠的债"
- 玩家的所有行为都是自发的，驱动力完全来自内驱
- 事后生成的清单条目可以附带意外惊喜（"原来这个也算！"）

**代表**：《黑帝斯》的预言清单很大程度上有这个特征——玩家可能在尝试各种武器组合后才发现"原来集齐某个系列的武器还有额外奖励"。

## 五、清单式游戏的深层设计

### 5.1 什么是"清单式游戏"

"清单式游戏"不是正式品类名，而是指**以"完成清单"为核心游戏机制**的一类游戏。它们的特征不是"有任务系统"，而是**整个游戏就是一张大清单**。

典型特征：
- 游戏内容 = 一组可完成的目标
- 核心循环 = 选择目标 → 执行 → 打勾 → 新目标解锁
- 进度 = 完成的条目数
- 终结感 = 清单全部完成

### 5.2 清单式游戏的子类型

| 子类型             | 核心机制                           | 代表作品                                 |
| :----------------- | :--------------------------------- | :--------------------------------------- |
| **清单式开放世界** | 地图标记兴趣点，逐个清理           | 《孤岛惊魂》系列、早期《刺客信条》       |
| **成绩单式**       | 挑战清单，每个条目是一个高难度目标 | 《以撒的结合》全成就、《杀戮尖塔》进阶20 |
| **Bingo式**        | 多条任务线并行，完成一条划一条     | 独立游戏中常见的"To-Do List"机制         |
| **渐进解锁式**     | 清单 = 内容解锁路线图              | 模拟经营游戏的"科技树"、手游的"战役地图" |

> ⚠️ **注意区分**："有收集要素"不等于"清单式游戏"。清单式游戏的核心特征是**清单本身是驱动力的主要来源**。如果一个游戏的收集要素是玩家自发探索的副产物（如《旷野之息》的呀哈哈），它用的是**记录式清单**——告诉你"有什么、完成了什么"，而非"你必须做什么"。这一点在§5.5展开。

### 5.3 清单式游戏的体验陷阱

| 陷阱               | 表现                       | 成因                       | 对策                                   |
| :----------------- | :------------------------- | :------------------------- | :------------------------------------- |
| **"清单疲劳"**     | 玩家看到大量标记点后放弃   | 清单密度过高，缺乏节奏控制 | 渐进式揭露清单内容，而非一次性全部显示 |
| **"为清而清"**     | 玩家变成机械执行，体验空洞 | 清单条目间缺乏差异性       | 每个条目应有独特的微叙事/小挑战        |
| **"错过焦虑"**     | 担心遗漏内容，产生错过焦虑 | 清单条目是隐藏的/限时的    | 明确告知哪些是可错过的、哪些会保留     |
| **"通关即弃"**     | 清单清完后毫无留恋         | 清单完成后无新的体验       | 清单完成应有仪式感/质变                |
| **"最优路径固化"** | 玩家查攻略按最优顺序清图   | 清单设计没有动态反馈       | 清单条目的难度/奖励随玩家顺序变化      |

### 5.4 核心概念纠偏：规定式 vs 记录式清单

这是整个文档最重要的概念区分。**把《旷野之息》《黑帝斯》《星露谷物语》简单归为"清单式游戏"是概念错误**，需要先澄清：

| 维度             | 🚫 规定式清单                      | ✅ 记录式清单                 |
| :--------------- | :-------------------------------- | :--------------------------- |
| **心理模型**     | "系统要我做什么"                  | "我发现有什么可做的"         |
| **驱动力来源**   | 外驱——完成任务获得奖励            | 内驱——好奇心与自主目标驱动   |
| **清单推送方式** | 主动推送（弹窗/提醒/红点）        | 被动查阅（玩家主动打开查看） |
| **条目呈现**     | 一次性全部列出已完成/未完成       | 渐进浮现，发现一条记录一条   |
| **完成压力**     | 高——未完成条目持续可见产生焦虑    | 低——条目是"记录"而非"命令"   |
| **主线影响**     | 不完成可能卡主线进度              | 完全可选，不影响核心推进     |
| **典型代表**     | 育碧式开放世界、大部分MMO任务日志 | 旷野之息、黑帝斯、黑暗之魂   |

**关键洞察**：这三款游戏能成为典范，不是因为它们的"清单设计做得好"，而是因为它们**刻意让玩家不觉得自己在面对一张清单**：

- **《旷野之息》**：呀哈哈和神庙完全没有清单界面——玩家是自己在地图上标记、自己决定追不追求100%。游戏从不出"你还有X个没找到"的提示。它是**反清单的设计**。
- **《黑帝斯》**：预言清单在后台渐进解锁，玩家获得新内容时"哦原来还有这个"——清单是**惊喜的载体**而非待办事项。它用了大量隐藏/条件解锁让清单本身成为探索奖励。
- **《星露谷物语》**：社区中心收集包有一定的清单特征，但游戏的核心体验是"我今天想做什么"而非"我必须完成什么"。收集包只是众多可能目标之一，且玩家可以完全跳过它走Joja路线。

这些游戏的共同秘密：**将清单的底层心理机制（目标梯度效应/蔡格尼克效应）嵌入到玩家自主探索的框架中，让玩家感觉"是我要做的"而非"它要我做的"。**

### 5.5 真正的清单式正面设计案例

以下游戏才是**在显式清单框架下做出了好体验**的正面案例——它们不回避"这就是一张清单"的事实，但通过设计让清单本身变得有趣：

| 游戏                        | 清单形式                         | 为什么好                                                              | 可学之处                                       |
| :-------------------------- | :------------------------------- | :-------------------------------------------------------------------- | :--------------------------------------------- |
| **《死亡搁浅》**            | 订单清单+区域连通度进度          | 每个订单之间有地理/剧情的连贯性，清单条目串联成"建造运输网络"的大叙事 | 清单条目之间不是孤立的，彼此构成叙事和空间逻辑 |
| **《中土世界：暗影魔多》**  | 复仇系统→兽人队长猎杀清单        | 清单条目是**动态生成的**，每局不同，且与玩家的过往行为直接相关        | 清单因人而异，玩家对"我的清单"有所有权感       |
| **《如龙》系列**            | 支线任务清单+小游戏清单+收集清单 | 清单条目各自是独立的迷你玩法，并非"去A处杀N只怪"的同质化填空          | 每个清单元本身是好玩的，清单只是入口目录       |
| **《PowerWash Simulator》** | 清理区域清单                     | 清单与核心玩法完全绑定——每喷一块区域都是在打勾                        | 清单即玩法，打勾行为本身就是核心循环的一部分   |
| **《A Short Hike》**        | 登山清单（隐式）                 | 清单是探索的副产物而非主驱动力，完成后不产生"还得继续"的压力          | 清单数量少、节奏自然、完成后有满足感但无强迫感 |

## 六、清单设计自检清单（元清单）

> 设计一个任务/清单系统后，用以下条目自检：

### 6.1 基础结构
- [ ] 玩家能否一眼看出当前有多少任务可做？
- [ ] 清单中的条目是否按优先级/区域/类型有结构分组？
- [ ] 是否有"已完成"的归档视图，让玩家回顾成就？
- [ ] 玩家能否主动放弃/删除不感兴趣的任务？

### 6.2 节奏与密度
- [ ] 同时给出的任务数量是否控制在 3-7 项？
- [ ] 是否有主线/支线的明确区分？
- [ ] 玩家是否偶尔有"没有任务"的留白时刻？（而非一直高密度）
- [ ] 清单最末端是否有令人期待的"终极目标"？

### 6.3 奖励与反馈
- [ ] 每个任务完成后是否有即时的正反馈信号？
- [ ] 多个任务完成后是否有累积性的质变奖励？
- [ ] 任务奖励是否与玩家当前的需求匹配？（非无脑给用不上的东西）
- [ ] 清单的进度是否在任何时候都清晰可见？

### 6.4 自主感保护
- [ ] 玩家是否可以自主选择任务顺序？（非强迫线性）
- [ ] 是否有多条路径/多种方式完成同一个任务？
- [ ] 清单是否包含**可选的高难度目标**（额外挑战）？
- [ ] 玩家是否在"我想做"而非"系统要我做什么"的状态？

### 6.5 去目标化平衡
- [ ] 除了任务清单外，是否有其他引导方式（视觉/环境/好奇心引力）？
- [ ] 清单是否只是**辅助工具**而非唯一的驱动力？
- [ ] 玩家不看清单也能自然地推进游戏吗？
- [ ] 清单条目的位置是否通过**自然探索**发现，而非全部标记在地图上？

### 7.2 与框架设计的关系

清单系统本身可以作为**一个独立的框架**来设计：

```
清单框架 = 任务生成机制 + 进度追踪 + 奖励投递 + 条目多样性
```

在框架设计原则中（参考 `技能/专业/框架设计.skill.md`）：
- **奥卡姆剃刀**：清单中的每一条都必须问"移除它，玩家失去什么？"
- **快乐密度**：清单条目的数量和难度应与快乐密度曲线匹配，避免"纯跑腿"稀释密度
- **驱动力优先**：清单应主要激发内驱（"我想完成"），而非只依赖外驱（"完成了给我东西"）

### 7.3 可与清单设计结合的其他知识库空白方向

清单设计是**任务系统**的子集。如果你未来要补 `任务与使命设计` 这个空白方向（从上一轮分析中识别的），清单设计可作为那篇文章的§三基础。

## 八、总结

### 核心理念一句话

> **规定式清单的本质是"给玩家安排任务"，记录式清单的本质是"帮玩家记录旅程"。你需要清楚自己设计的是哪一种——前者赌的是外驱力，后者赌的是你的内容够不够吸引人。**

### 三条不可违背的原则

1. **清单是辅助，不是主宰**——玩家应能在不看清单的情况下自然推进游戏。如果玩家90%的时间在看清单而非观察世界，你的清单已经越界了。
2. **规定式要克制，记录式要隐藏**——规定式清单应严格限制条目数量和推送频率（上限7条）；记录式清单应让玩家主动去翻开，而非弹窗"你还有3个未发现！"
3. **打勾本身应是快乐，而非解脱**——如果玩家完成任务后的第一感觉是"总算不用再看这条了"而非"有意思，下一个是什么？"，你的清单已经沦为负担。

### 开放性问题（待进一步研究）

- 清单密度与心流通道的关系：清单过密 → 焦虑；清单过疏 → 无聊；最佳密度如何量化？
- 清单条目的"微叙事"设计：每个任务在10秒内是否能传达一个有趣的情境？
- 动态/程序化生成的清单（如《暗影魔多》的复仇系统）：能否在不牺牲质量的前提下规模化？
- 多人游戏中的共享清单（公会任务/世界事件）：协作清单 vs 个人清单的设计差异？
