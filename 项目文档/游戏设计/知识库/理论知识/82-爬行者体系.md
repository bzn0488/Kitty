# 爬行者体系——即时正反馈循环设计模型

> 标签：`decision-design`, `feedback-loop`, `player-psychology`, `game-mechanics`, `happiness-total`
> 关联概念：`79-相对最优解与三层变化模型.md`, `46-正反馈与负反馈循环.md`, `43-游戏节奏设计.md`, `27-快乐总量.md`, `10-快车道设计.md`, `29-体验图分析框架.md`

## 一、定义

**爬行者体系**（Crawler System）是一种以**即时正反馈循环**为核心的游戏设计模型。其基本逻辑为：

```
玩家做出简单决策 → 即时给予正反馈 → 强化决策行为 → 继续决策 → 循环
```

与**相对最优解体系**（强调决策质量）和**绝对最优解体系**（强调找到标准答案）不同，爬行者体系**强调决策的数量和频率**——决策越快、越密集，获得的正反馈总量越多。

> **本质**：爬行者体系是一种**决策框架（Frame）的设计模型**，而非单纯的反馈循环模型。它的核心是为玩家提供清晰的**思考基点（Anchor Point）**——玩家在框架内可以快速、连续做出决策并获得反馈。

## 二、核心机制

### 2.1 即时反馈

每一条决策都必须立即产生可感知的反馈。在爬行阶段内，反馈可以是正反馈也可以是负反馈——负反馈同样作为**信号**指导玩家在框架内调整行为。

```
决策 → 即时反馈（正/负） → 调整决策 → 继续循环
```

> ⚠️ **概念校正**：初版将爬行者体系理解为"只存在正反馈"，这一定义过窄。爬行者体系的本质是**决策框架**，框架内的反馈可以是正（确认做对了）也可以是负（提示需要调整），关键在于反馈是**即时且可感知的**。负反馈在此不是惩罚，而是信号——例如"按费用递增出牌，间隔1时翻倍"的设计中，打错顺序效果减半即是框架内的负反馈信号，引导玩家优化出牌顺序。

### 2.2 简单决策

爬行者体系中的"简单"不是指决策本身容易，而是指决策有明确的**思考基点（Anchor Point）**——玩家一眼就能识别、不假思索即可执行的决策规则。

- 基于**希克定律**：选项数量应严格限制，避免决策疲劳
- 提供清晰的**思考基点**，避免玩家"理线头"——即找不到从何开始思考
- 决策难度应当**下降**而非上升，以维持正反馈循环的速度

> **思考基点**是通用设计概念，指玩家在面临选择时从何处开始思考的初始锚点。它不局限于爬行者体系，在所有决策设计中都存在（如叙事选择中的"角色性格基点"、解谜中的"线索基点"）。爬行者体系是思考基点的典型应用场景——在快速连续决策中，思考基点的清晰度直接决定了决策效率。

### 2.3 快乐总量公式

在一段连续的爬行者体验中：

```
快乐总量 ≈ 正反馈强度 × 决策频率
```

由于正反馈强度相对恒定（都是即时小奖励），**决策频率**成为影响快乐总量的关键变量。玩家会自然调节决策速度——发现自己获得反馈越快，就越愿意加快决策节奏。

## 三、框架 vs 非框架：爬行者体系的双层结构

爬行者体系的本质是**决策框架设计**，而非单一的反馈循环模型。需要区分两个层面：

### 3.1 框架层（Frame Layer）

框架层是爬行者体系提供的**决策规则和边界**，定义了玩家"可以做什么"和"应该怎么做"：

| 组件         | 说明                     | 案例                 |
| :----------- | :----------------------- | :------------------- |
| **思考基点** | 从何处开始决策的初始锚点 | "按费用从低到高出牌" |
| **决策序列** | 决策的先后顺序或优先级   | 1费→2费→3费          |
| **规则边界** | 哪些做法的哪些行为有效   | 间隔为1时效果翻倍    |
| **反馈规则** | 如何告知玩家做对/做错了  | 满足条件→翻倍动画    |

### 3.2 执行层（Execution Layer）

执行层是玩家在框架内实际进行的操作和获得的反馈：

```
框架层：定义了"按费用递增出牌，间隔1则翻倍"
执行层：玩家打出2费牌 → 系统确认1费已打 → 标记为"间隔1" → 打出3费 → 触发翻倍
```

### 3.3 为什么需要双层区分

- **框架层不变**：在整个爬行阶段内，框架保持稳定——玩家不需要重新学习"如何决策"
- **执行层可变**：每一轮的实际决策内容可以不同（对手牌不同），但决策方式一致
- **学习成本低**：玩家只学一次框架，后续重复使用

这种双层设计正是爬行者体系区别于"一堆随机事件"的关键——它不是让玩家一直做新鲜决策，而是**在固定框架内做重复但不同的决策**。

> **类比**：节奏游戏（如《节奏大师》）的框架是"音符从上方落下，到了判定线按对应键位"——玩家只学一次，每首歌都是在这个框架内执行。爬行者体系同理。

### 3.4 框架的不可变性与阶段切换

爬行者框架在**一个阶段内**应当保持稳定。当框架本身需要变化时，意味着进入了新的阶段：

```
阶段1（框架A：费用递增出牌）→ 阶段2（框架B：按阵营布阵）→ 阶段3（框架C：...）
```

框架变化本身即是**阶段切换信号**——玩家感知到"现在规则变了"，自动切换决策模式。

## 四、对比：三大决策体系

| 维度           | 爬行者体系                     | 相对最优解体系          | 绝对最优解体系     |
| :------------- | :----------------------------- | :---------------------- | :----------------- |
| **核心驱动力** | 快速获得正反馈                 | 智力优越感              | 找到标准答案       |
| **决策频率**   | 🔴 高（越快越好）               | 🟡 中（需思考）          | 🟢 低（一次找对）   |
| **反馈强度**   | 🟢 小但密集                     | 🔴 大但稀疏              | —                  |
| **随机性需求** | 低                             | 高（体验图扩展）        | 无                 |
| **选项数量**   | 少（希克定律）                 | 中等（需权衡）          | 固定               |
| **玩家状态**   | 操作→反馈→操作                 | 思考→决策→反馈          | 识别→执行          |
| **典型代表**   | 割草游戏、点击器、部分放置游戏 | Roguelike DBG、策略游戏 | 解谜游戏、传统塔防 |

## 五、设计要点

### 5.1 阶段化设计

纯爬行者体系存在两个天然问题：
1. **玩家疲惫**——长期高频率操作导致手/脑疲劳
2. **边际递减**——相同正反馈反复给予，效果逐渐减弱

**解决方案**：划分阶段或设立明确的**体验节点**。

```
阶段1（爬行期）→ 体验节点（目标达成）→ 阶段2（爬行期）→ ...
```

每个阶段内部保持爬行者节奏，阶段之间通过目标感让玩家获得喘息和新的方向。

### 5.2 选项限制

基于希克定律（Hick's Law），选项数量与决策时间呈对数关系：

| 选项数 | 相对决策时间 |
| :----: | :----------: |
|   2    |      1×      |
|   3    |     1.6×     |
|   4    |      2×      |
|   5    |     2.3×     |
|   6    |     2.6×     |

爬行者体系中建议选项数 ≤ 4，以保证决策节奏不被打破。

### 5.3 思考基点设计

爬行者体系需要通过设计明确的**思考基点**来降低玩家的决策启动成本。思考基点是玩家从"看到选项"到"开始决策"之间的最短路径。

#### 典型设计手法

| 手法         | 说明                                 | 示例                                                      |
| :----------- | :----------------------------------- | :-------------------------------------------------------- |
| **序列化**   | 按明确序列排序，玩家按序决策         | 按费用从低到高出牌——从 1 费打到 3 费，间隔为 1 时效果翻倍 |
| **维度凸显** | 每个选项突出一个核心维度             | 三选一中 A=攻击型 / B=防御型 / C=功能型                   |
| **差异对比** | 高亮选项间的关键差异点               | 装备对比中标注"推荐/当前/新获得"的差异                    |
| **视觉引导** | 用动效/放大/高亮指出最容易切入的选项 | 有足够资源使用的卡牌高亮边框                              |

> **序列化**是爬行者体系中最强的思考基点设计——玩家不需要思考"先做什么"，序列本身就是答案。上文的"按费用递增出牌，间隔1时效果翻倍"就是典型的序列化思考基点：玩家每次出牌只需确认当前费用的牌是否最合适，无需在全部手牌中搜索最优解。

## 八、设计案例

### 8.1 暗黑破坏神系列（刷宝循环）

```
击杀小怪 → 掉落光效/音效 → 拾取 → 继续寻找下一群小怪
```

每个击杀都是一次简单决策（用什么技能打哪个怪），掉落立即给予正反馈，决策频率极高。

### 8.2 Vampire Survivors

```
移动角色 → 怪物自动被范围技能击杀 → 经验值自动拾取 → 升级 → 三选一强化
```

玩家只需要做"移动"这一个简单决策，所有反馈即时且密集。

### 8.3 增量游戏（Cookie Clicker）

```
点击 → 数字增长（正反馈）→ 攒够数量 → 购买升级 → 点击效率更高
```

最简单的决策（点击）换取最密集的正反馈（数字跳动）。

### 8.4 增量自走棋：费用递增出牌

```
手牌按费用排序（1/2/3费）→ 按间隔1从低到高打出 → 若满足则效果翻倍 → 继续决策
```

这是一个典型的爬行者框架设计：
1. **框架明确**：出牌顺序由费用序列决定，玩家不需要"规划"策略
2. **决策可执行**：玩家只需在每张卡牌判定"当前是不是费用+1"，无需全局搜索
3. **结果有差异**：翻倍 vs 不翻倍构成了框架内的反馈差异
4. **思考基点清晰**：费用递增序列就是思考基点——玩家一看就懂从哪开始

此设计在增量自走棋中适用于**前期快速开包排序、商店刷新决策**等快速连续决策场景；在**核心Build方向抉择、Boss战策略规划**等需要深度思考的场景则切换至相对最优解模式。

## 九、延伸思考

### 9.1 爬行者体系的后疲劳期

长时间爬行者体验后，玩家必然进入"正反馈疲劳期"。此时需要：
- 切换至**相对最优解模式**（需要思考的决策）
- 或引入**负反馈**（挑战/风险）重新激活感知
- 或**结束当前阶段**，让玩家进入新的体验节点

### 9.2 混合设计

成熟游戏通常是两种体系的混合：

```
阶段A：爬行者（快速成长、积累资源）
     ↓
阶段B：相对最优解（关键 Build 决策、BOSS 战）
     ↓
阶段C：爬行者（验证 Build、收割反馈）
```
