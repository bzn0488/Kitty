# 涌现式与系统交互设计

> 整理自：`_原始文档归档/塞尔达&涌现式.txt` + `父系母系.txt`（系统交互部分）

## 一、涌现式设计

### 1.1 定义
让游戏内的系统之间有所联系。通过多个系统的连锁反应产生意料之外、情理之中的效果。

**经典例**（矮人要塞）：
> 猫死了 → 酒精中毒 → 舔毛时舔到酒精 → 被人打架打翻的酒瓶溅到 → 矮人脾气暴躁会打架

每个环节都是一个独立系统，缺一个就不会发生。

### 1.2 涌现式设计的制作流程

| 方法 | 步骤 | 代表 | 特点 |
|------|------|------|------|
| **先搭建系统再修剪** | 搭建物理系统→编写规则→崩溃效果→裁剪 | NOITA | 牵一发而动全身，难判断取舍 |
| **追随乐趣（Follow the Fun）** | 提想法→做原型→游玩→扩有趣部分 | 任天堂（马里奥、塞尔达） | 成本难控，但效果明显 |

千水观点：追随乐趣的方式**强于**预制系统再修剪。因为在一个已建立的系统框架下做减法，很难判断决定是否值得。

### 1.3 一个操作有多种用处
任天堂核心理念：让一个操作在不同情况下有多种效果。

**例**：马里奥的跳跃
- 小马里奥→踩死蘑菇怪
- 大马里奥→顶碎方块
- 吃了花的马里奥→斜下方45度子弹更好命中

**例**：塞尔达中的火焰
- 燃烧草地
- 烤肉
- 产生上升气流

## 二、涌现式系统结构

### 2.1 传统结构（差）
多个系统为一个系统/体验服务：
```
锻造系统 → 额外攻击力
特性系统 → 战斗被动          → 最终战斗系统
升级加点 → 基础数值
```

### 2.2 一般涌现式结构（好）
一个系统可以和多个系统串联。

### 2.3 塞尔达式结构（最优）
不仅多了系统关联，还增加了**表现层**（如温度计）。

**例**：
- 踩岩浆掉血 → 因为温度升高，林克承受不了高温（表现层逻辑）
- 雪山时可以装备火属性武器让自己不寒冷

★ **玩家可以通过不同方法达成同一目的** → 引发玩家按自己方式解决问题

### 2.4 减少数值比重
当数值比重过大时，玩家会因追随数值而采用最优解，"一招鲜吃遍天"。

## 三、系统交互的局限

### 3.1 高度耦合
系统间高度关联，调整困难。

### 3.2 虚构层统一
不同系统的虚构层需要统一。规则不一致消磨严肃性。

**例**：岩石被设定为坚硬度3，锤子可以破坏坚硬度3以下障碍物。之后设计者用"岩石"视觉元素作为不能通行的区域——玩家尝试用锤子破坏，失败→尝试→怀疑→不信任。

## 四、母系游戏中的系统设计

### 4.1 系统间相互影响创造体验
当一个机制改变，其他机制也经由复杂关系改变，使不同行为的驱动力变化。

**形式**：
- **外因改变**：天气、版本变更等
- **体验产出移动**：体验模块的最终结果产生位移，改变与其他模块的距离
- **多个系统产出互为需求**：玩家自发产生不同目标
- **渐进式生成**：根据玩家行为生成新体验模块

### 4.2 母系思维改良方向
**通用机制→非通用机制转变**：使特定势力/个体得以凸显，避免信息量过大或区分度过小。

- **宏观层**：通用的普适机制（保证基本运行）
- **微观层**：表现特定势力/个体的行为（替换/填充在宏观层之上）

**例**（怪物猎人）：
- 宏观层：大多数怪物血量低会逃跑→休息
- 微观层：某些亚种临死反而更亢奋
