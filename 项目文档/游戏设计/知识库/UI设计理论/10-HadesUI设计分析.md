# Hades UI 设计分析

> 《Hades》由 Supergiant Games 开发，其 UI 设计的核心成就在于：**在高速动作游戏中，让大量信息（Build构筑、角色对话、进度追踪）以极低的认知摩擦呈现给玩家。**

---

## 一、核心设计哲学：信息密度与节奏的平衡

Hades 面临一个独特的 UI 挑战——作为一款 Roguelite 动作游戏，它需要在**高速战斗中保持清晰**，同时承载**剧情叙事**和**Build 构筑**两大重量级系统。

### 1.1 "三口"式信息架构

Hades 将所有交互流程归纳为三个"口"：

```
┌──────────┐    ┌──────────┐    ┌──────────┐
│  战斗口   │ → │  选择口   │ → │  叙事口   │
│ 高速动作  │    │ Build决策 │    │ 角色互动  │
│ 即时反馈  │    │ 恩赐选择  │    │ 对话/赠礼 │
└──────────┘    └──────────┘    └──────────┘
```

**关键的 UI 洞察**：每个"口"使用完全**不同的交互模式和信息密度**，玩家大脑在不同模式间切换时获得自然的节奏变化。

| 口   | 交互模式 | 信息密度             | 输入设备       |
| :--- | :------- | :------------------- | :------------- |
| 战斗 | 实时动作 | 极低（即时反馈为主） | 攻击/闪避/投弹 |
| 选择 | 暂停决策 | 高（多选一的权衡）   | 方向键选择     |
| 叙事 | 对话交互 | 中（文字+情感）      | 对话选项       |

> **与知识库关联**：三口式信息架构本质上是"游戏节奏设计"中的紧张-松弛模式在 UI 层面的映射。——`理论知识/43-游戏节奏设计.md`

## 二、战斗 UI — 在高速中保持清晰

### 2.1 极简战斗 HUD

Hades 的战斗 HUD 极简到只有绝对必要的元素：

```
┌──────────────────────────────────────┐
│  左上: 血量条（绿色→黄色→红色）        │
│  左下: 特殊槽（神力的蓄力状态）         │
│  右下: 投弹指示器（剩余投弹数+冷却）    │
│  底部: 快捷键指示器（召唤/道具）        │
│                                      │
│  ★ 无经验条                           │
│  ★ 无等级显示                         │
│  ★ 无小地图                           │
│  ★ 无任务提示                         │
│  ★ 无伤害数字（默认关闭）              │
└──────────────────────────────────────┘
```

### 2.2 敌我状态呈现

| 技术           | 实现                        | 效果                         |
| :------------- | :-------------------------- | :--------------------------- |
| **敌人血量条** | 敌人头顶细条，仅可见时显示  | 不遮挡视线，信息按需呈现     |
| **精英标记**   | 精英敌人有金色边框+名称显示 | 一眼区分敌人类别             |
| **Boss 血条**  | 屏幕顶部大型血条+阶段标记   | Boss 战专用 UI               |
| **护甲/护盾**  | 白色额外血量层在血条上方    | 清晰区分不同类型生命值       |
| **状态图标**   | 角色头顶/身上特效+图标      | 中毒/虚弱/强化等状态直观呈现 |

### 2.3 战斗反馈的"Juiciness"

Hades 的战斗反馈密度极高，贯穿了"每一次操作都有意义"的理念：

| 反馈类型     | 具体实现                                                     |
| :----------- | :----------------------------------------------------------- |
| **击中反馈** | 敌人闪烁 + 命中特效粒子 + 屏幕震动 + 音效 + 慢动作（特殊时） |
| **闪避反馈** | 角色虚化 + 烟雾粒子 + 成功闪避时的时间膨胀效果               |
| **神力槽满** | 角色发光 + 屏幕闪烁 + 音效变化                               |
| **击杀反馈** | 敌人碎裂/化为灰烬 + 掉落的收集物闪光                         |
| **受击反馈** | 屏幕边缘红色闪烁 + 角色后仰 + 音效                           |

> **与知识库关联**：「交互反馈与动效设计」中的反馈链路模型——Hades 的每次攻击都触发了完整的"即时确认→结果反馈→状态反馈"链路。——`UI设计理论/03-交互反馈与动效设计.md`

## 三、恩赐选择界面 — Build 决策的 UX 设计

### 3.1 三选一交互

Hades 的核心 Build 机制——恩赐选择——使用三选一模式：

```
┌──────────────────────────────────────┐
│                                      │
│         [宙斯] 赐予你力量              │
│                                      │
│   ┌──────────┐ ┌──────────┐ ┌────────┐│
│   │ 雷霆打击  │ │ 风暴突进  │ │ 闪电子弹││
│   │ 攻击触发  │ │ 冲刺带电  │ │ 投弹链  ││
│   │ 连锁闪电  │ │ 击退效果  │ │ 自动追  ││
│   │          │ │          │ │ 踪     ││
│   └──────────┘ └──────────┘ └────────┘│
│                                      │
│  已拥有: 雷霆爆发                     │
│  推荐: ★★★★☆                          │
└──────────────────────────────────────┘
```

### 3.2 UX 设计要点

| 设计点           | 分析                                           |
| :--------------- | :--------------------------------------------- |
| **信息卡片化**   | 每个恩赐以独立卡片呈现，包含图标+名称+效果简述 |
| **效果预览**     | 选中后可在"预览模式"查看详细数值               |
| **已有效果标注** | 已有类似效果时显示"已拥有"，避免重复选择       |
| **稀有度标识**   | 普通/稀有/史诗/英雄的边框+光效区分             |
| **Build 信号**   | 推荐等级（星星）引导新玩家，但不限制老玩家     |
| **无时间压力**   | 选择界面完全暂停，玩家有充足时间权衡           |

> **与知识库关联**：「三选一优化原理」在 Hades 中得到了最充分的实践——三个选项各有侧重，让玩家每次选择都是有趣的权衡。——`理论知识/16-卡牌设计规范.md`

### 3.3 房间奖励的"预览"UI

Hades 中的房间奖励通过**房间门口的图标**提前告知，这是"信息前置"设计的典范：

| 图标             | 含义                   | UX 价值                           |
| :--------------- | :--------------------- | :-------------------------------- |
| 奥林匹斯神的头像 | 恩赐奖励               | 玩家可针对 Build 需求选择神祇路线 |
| 金币袋           | 金币奖励               | 可用于商店消费                    |
| 心形             | 生命值上限提升         | 生存路线选择                      |
| 锤子             | 武器改造（赫菲斯托斯） | 核心 Build 变化                   |
| 钥匙             | 解锁武器/天赋点数      | 长期成长                          |

**设计洞察**：玩家在进入房间**之前**就知道奖励类型，这使他们能做路线规划，而非被动接受。这是"玩家自主权"在 UI 层面的体现。

> **与知识库关联**：「权力下放设计」——信息前置让玩家感觉自己掌控着路线规划。——`理论知识/22-权力下放设计.md`

## 四、大厅（Hub）UI——叙事与进度的一体化

### 4.1 哈迪斯大堂的 UI 融入

Hades 的大堂（Hub）将大量 UI 功能融入场景中：

| 功能     | 场景位置         | UX 效果                         |
| :------- | :--------------- | :------------------------------ |
| 武器选择 | 大厅中央武器架   | 切换武器同时看到角色换装        |
| 能力升级 | 镜子（夜之圣镜） | Diegetic UI——通过游戏内物体升级 |
| 对话     | 与 NPC 交互      | 所有叙事通过角色对话呈现        |
| 商品购买 | 商店柜台         | 交易在场景中完成                |
| 资源查看 | 大厅各处容器     | 资源收集直接关联场景            |

### 4.2 死亡后的"惩罚"UI

Hades 中死亡后的界面设计巧妙地将负面体验转化为正向动机：

- **死亡动画** → 节奏切换信号
- **结算界面** → 显示本次获得的资源（化负为正）
- **暗影之镜** → 看到永久升级的进步
- **NPC 对话** → 推进剧情，让死亡有意义
- **新的对话** → 下次死亡前未见的剧情

> **与知识库关联**：「正反馈与负反馈循环」——Hades 将死亡（负反馈）转化为剧情推进（正反馈），UI 在其中扮演了关键角色。——`理论知识/46-正反馈与负反馈循环.md`

## 五、渐进式 UI 披露——随玩家成长

### 5.1 信息逐步开放的策略

Hades 的 UI 会随玩家进程逐步开放新功能：

| 进程节点           | 开放内容                     |
| :----------------- | :--------------------------- |
| 第一次死亡         | 解锁大厅 + 暗影之镜升级系统  |
| 首次击败第一个Boss | 解锁武器形态切换             |
| 收集 X 个钥匙      | 解锁武器升级面板（泰坦之血） |
| 推进剧情           | 解锁伴侣/召唤系统            |
| 通关一次           | 解锁热度系统（难度调节）     |

### 5.2 设计原理

**"每一局都有新东西"**——即使玩家输了，UI 也会展示新解锁的内容，确保每次返回大厅都有变化：

- 新对话标记（NPC 头顶 "!"）
- 新武器形态解锁的发光特效
- 新天赋可用的醒目提示
- 新房间/区域的解锁

> **与知识库关联**：「学习与遗忘曲线」——渐进式 UI 披露确保玩家在每个阶段只处理适合当前认知水平的信息量。——`理论知识/39-学习与遗忘曲线.md`

## 六、UI 与 Roguelite 循环的配合

### 6.1 每局 UI 的一致性

Hades 中每局游戏的 UI 结构完全一致——无论玩家选择什么武器、Build 如何：
- **战斗 HUD** 始终相同位置
- **恩赐选择界面** 始终三选一
- **房间奖励预览** 始终在门口

**价值**：一致性减少了学习成本，让玩家专注于 Build 策略而非"这个 UI 怎么操作"。

### 6.2 每局的差异感

尽管 UI 一致，Hades 通过**视觉变化**传达每局的不同：
- 不同武器的战斗特效差异
- 不同神祇的恩赐配色和图标
- 不同房间奖励的图标和光效
- 热度系统的紫色界膜视觉效果
