# 游戏 UX 评估方法

> 知识库既有大量设计理论，但缺少"如何评估设计质量"的方法论。本文档补齐这一环——评估不是事后检查，而是设计流程中的核心环节。

---

## 一、游戏启发式评估法

### 1.1 什么是启发式评估

启发式评估是评估者对照一组已知的可用性启发式（最佳实践检查表）来审查游戏界面，标记违反项。优点是**低成本、快速、不需要真实玩家**；缺点是无法替代真实用户测试。

### 1.2 游戏专用启发式清单

以下启发式综合了 Nielsen 10 项可用性启发式与游戏领域特有原则：

|   #   | 启发式                           | 说明                                               | 对应游戏原则 |
| :---: | :------------------------------- | :------------------------------------------------- | :----------- |
|  H1   | **系统状态可见性**               | 玩家应始终知道"我在哪、我在做什么、系统状态是什么" | HUD + 反馈   |
|  H2   | **系统与真实世界的匹配**         | 使用玩家熟悉的隐喻和语言，而非技术术语             | 术语一致性   |
|  H3   | **用户控制与自由**               | 玩家应能撤销/重做操作，有退出路径                  | 容错性       |
|  H4   | **一致性与标准**                 | 同一概念在不同场景中表现一致                       | 一致性       |
|  H5   | **错误预防**                     | 设计应防止错误发生，而非仅提供好用的错误提示       | 容错性       |
|  H6   | **识别而非回忆**                 | 选项应可见，玩家不需要记住信息                     | 可见性       |
|  H7   | **灵活性与效率**                 | 提供快捷键、自定义选项供高级玩家使用               | 效率         |
|  H8   | **美观与简约设计**               | 不相关或罕见的信息应隐藏或折叠                     | 视觉层次     |
|  H9   | **帮助玩家识别、诊断和恢复错误** | 错误信息应清晰说明问题与解决方法                   | 容错性       |
|  H10  | **帮助与文档**                   | 提供易于搜索的教程和帮助系统                       | 新手引导     |

### 1.3 游戏补充启发式

|   #   | 启发式                   | 说明                                   |
| :---: | :----------------------- | :------------------------------------- |
|  G1   | **操作反馈一致性**       | 所有玩家操作应在 < 100ms 内得到确认    |
|  G2   | **核心玩法学习的渐进性** | 教学不应一次倾泻，应逐步引入           |
|  G3   | **HUD 信息相关性**       | 屏幕上每项信息都应有明确用途           |
|  G4   | **情感反馈**             | 游戏应对正/负面结果给予恰当的情感回报  |
|  G5   | **控制映射自然性**       | 操作映射应符合直觉（如方向键控制移动） |
|  G6   | **失败恢复**             | 失败不应带来过重的惩罚，提供重试路径   |

> **与知识库关联**：评估是「反思设计法」六步流程中的核心环节。——`理论知识/06-反思设计法.md`

## 二、认知走查法（Cognitive Walkthrough）

### 2.1 适用场景

认知走查适合评估**新手玩家的首次操作体验**——评估玩家是否能在无外部帮助下完成任务。

### 2.2 四步走查框架

评估者针对每个任务步骤回答以下四个问题：

|   #   | 问题                             | 评估标准                       |
| :---: | :------------------------------- | :----------------------------- |
|  Q1   | **玩家能否意识到这个操作可行？** | 控制/UI 元素是否可见、可识别？ |
|  Q2   | **玩家是否会尝试这个操作？**     | 操作与目标的关联是否明显？     |
|  Q3   | **玩家能否找到正确操作方式？**   | 操作方式是否符合玩家预期？     |
|  Q4   | **玩家能否从反馈得知操作成功？** | 即时反馈是否清晰、明确？       |

### 2.3 游戏专用走查变体

对于游戏，新增两条问题：

|   #   | 问题                             | 说明                     |
| :---: | :------------------------------- | :----------------------- |
|  Q5   | **操作结果是否符合玩家预期？**   | 反馈是否与系统规则一致？ |
|  Q6   | **失败后玩家是否知道如何调整？** | 玩家能否从失败中学习？   |

> **与知识库关联**：「学习与遗忘曲线」解释了为什么新手引导期的评估（认知走查）比精通期的评估更重要。——`理论知识/39-学习与遗忘曲线.md`

## 三、可用性测试（Usability Testing）

### 3.1 测试类型

| 类型           | 时机     | 方式              | 产出             |
| :------------- | :------- | :---------------- | :--------------- |
| **形成性测试** | 设计早期 | 低保真原型 + 观察 | 定性设计方向修正 |
| **总结性测试** | 开发后期 | 高保真原型 + 度量 | 定量可用性指标   |
| **A/B 测试**   | 上线后   | 对照组实验        | 数据驱动的优化   |

### 3.2 度量指标

| 指标           | 计算方式                          | 含义           |
| :------------- | :-------------------------------- | :------------- |
| **任务完成率** | 成功完成任务的人数 / 总人数       | 界面可用性     |
| **任务用时**   | 完成任务的平均时间                | 操作效率       |
| **错误率**     | 错误操作次数 / 总操作次数         | 防错设计质量   |
| **首次点击率** | 第一步点击正确的比例              | 信息架构清晰度 |
| **SUS 分数**   | System Usability Scale 标准化问卷 | 综合可用性     |

### 3.3 游戏特定度量

| 指标              | 测量方式                                | 含义           |
| :---------------- | :-------------------------------------- | :------------- |
| **心流中断次数**  | 玩家在体验过程中主动查看菜单/教程的次数 | HUD 与导航效率 |
| **操作-反馈延迟** | 玩家操作与系统反馈之间的时间差          | Game Feel 质量 |
| **教程回看率**    | 玩家回看教程/帮助的频率                 | 教学效果       |

## 四、游戏 UX 评估流程

### 4.1 四阶段评估流程

```
设计阶段             开发阶段             测试阶段             上线后
   │                   │                   │                   │
   ▼                   ▼                   ▼                   ▼
启发式评估          认知走查            可用性测试           数据监控
+ 竞品分析          + 操作流分析         + 玩家观察           + 埋点分析
+ 设计评审          + 错误路径图         + 问卷调查           + 用户反馈
```

### 4.2 评估频率建议

| 阶段     | 频率     | 方法                       |
| :------- | :------- | :------------------------- |
| 每周     | 自查     | 启发式检查清单过一遍       |
| 每里程碑 | 深度评估 | 认知走查 + 内部可用性测试  |
| 每版本   | 外部测试 | 5-8 名外部玩家的可用性测试 |
| 上线后   | 持续     | 数据埋点 + 用户反馈收集    |

---

## 五、检查清单

- [ ] 设计完成后是否经过启发式评估（H1-H10 + G1-G6）？
- [ ] 是否对首次体验做过认知走查（Q1-Q6）？
- [ ] 关键路径是否经过可用性测试（至少 5 人）？
- [ ] 是否有量化指标（任务完成率/用时/错误率）定义可用性目标？
- [ ] 评估结果是否反馈到下一轮迭代中？
