## 核心定义

**框架→GDD 转换** = 将框架设计方案（系统定位+链接+原则+验证）扩展为完整的游戏设计文档（规则细节+数值+UI+叙事+美术+技术规格），供开发团队直接执行。

### 与框架设计的关系

```
框架设计 Skill             框架→GDD Skill
─────────────────          ─────────────────
设计"为什么做"        →     记录"怎么做"
定义系统定位          →     展开为具体规则
三原则审核            →     知识库验证+双层自检
输出框架方案          →     输出 GDD（可执行文档）
```

### 适用场景

| 场景                      | 说明                                                            |
| :------------------------ | :-------------------------------------------------------------- |
| **框架设计完成后**        | 有了成熟的框架方案系统清单和原则审核结果后，需要落地为开发文档  |
| **项目立项阶段**          | 需要向团队/投资方展示完整的游戏设计方案                         |
| **已有 GDD 需要补充系统** | 为现有 GDD 中的单个系统提供框架级重新设计（可跳过不相关的步骤） |
| **与 GDD 分析配合**       | 先用本 Skill 生成 GDD，再用 GDD 分析工作流反向验证              |

## 设计方法论

### 核心原则

1. **设计目的第一** — GDD 的每个系统章节必须以「设计目的」开头，执行者必须先理解"为什么做"，再读"怎么做"。任何与设计目的矛盾的规则都是错误的。
2. **框架不可违背** — GDD 是对框架的展开和具体化，不是重写。框架的核心原则（奥卡姆剃刀/快乐密度/驱动力优先）必须在 GDD 中得到忠实体现。
3. **外部参考辅助内部理论** — 框架和知识库提供设计原理支撑，联网搜索提供外部参考坐标（竞品/数值/风格/技术），两者互补。
4. **双层验证保障质量** — 第一层检查文档完整性/一致性/可执行性（写得对不对），第二层检查设计深度/冲突/体验路径/心理学风险（设计得好不好）。

### 工作流步骤

| 步骤         | 作用                                                  | 输入            | 输出                             |
| :----------- | :---------------------------------------------------- | :-------------- | :------------------------------- |
| ① 加载框架   | 理解框架设计的结构和内容                              | 框架设计方案    | 结构化框架摘要                   |
| ② GDD 大纲   | 规划 GDD 的章节结构                                   | 框架摘要        | 章节目录+来源标记                |
| ③ 系统展开   | 三阶段协作：建契约表→分组设计→回检+验证               | 大纲+知识库理论 | 接口契约表+系统章节+待办更新清单 |
| ④ 知识库验证 | 用理论验证每个系统的设计合理性                        | 系统章节        | 验证结果+待讨论项                |
| ⑤ 补充章节   | 设计叙事/美术/UI/技术等框架不覆盖的章节（含联网搜索） | 框架+系统章节   | 世界观/美术方案+UI要点           |
| ⑥ 用户确认   | 确认世界观方向、美术风格、待讨论问题                  | 上述所有输出    | 用户决策记录                     |

| ⑧ 双层自检   | 文档质量检查 + 设计深度分析                           | 生成的 GDD      | 自检报告+修复记录                |
| ⑨ 后处理     | 索引更新+结果通知                                     | —               | 完成确认                         |

### 系统展开的三阶段协作流程

框架转GDD 的核心步骤——系统展开——采用三阶段协作流程，在 AI 推进和用户决策之间来回迭代，确保跨系统连接无误。

#### 第一阶段：建接口契约表
在开始任何系统的详细设计前，先为所有系统列出接口契约——每个系统的名称、框架层的设计目的、依赖其他系统的接口、提供给其他系统的接口。契约表由用户审核确认后锁定。

#### 第二阶段：分组并行设计
系统按耦合度分组，每组一起设计而非逐个设计。典型分组模式：
- **第1组（骨架）**：与最多系统交互的核心组（如构筑+战斗）
- **第2组（经济）**：涉及同一资源流的组（如奖励+商店）
- **第3组（强度）**：共享判断逻辑的组（如难度+快车道）
- **第4组（局外）**：跨局运作的组（如规则变异+离场）
- **第5组（独立）**：依赖关系少的系统

设计过程中的跨组影响不立即回溯，而是记录为「待办更新」，等全部设计完成后再集中处理。

#### 协作回检节点
所有组设计完成后，呈现给用户：契约表、待办更新清单、各组系统设计摘要。用户逐条判断每条待办更新——采纳、拒绝或修改。

#### 第三阶段：修订+连接验证
根据用户决策处理待办更新。框架层修改退回框架设计；系统层修改更新契约表后重设受影响组；浅层修改直接在 GDD 中标注。完成后逐项验证每个系统的输入输出在契约表中是否匹配。

### 双层自检（步骤⑧）详解

> 这是本 Skill 区别于传统 GDD 编写的核心特征。

**Layer 1：文档质量**（12 项）
- 完整性（章节/系统/设计目的可见性）
- 一致性（系统引用/术语/框架对齐）
- 可执行性（规则可读/数值标记/决策标记）
- 框架原则对齐（奥卡姆剃刀/快乐密度/内驱优先）

**Layer 2：设计深度**（17 项）
- 冲突检测（意图/规则/资源/心流/时序）
- 体验路径断点（首次接触/早期/中期/精通期/引导）
- 心理学风险（过度理由/习得性无助/损失厌恶/认知过载）
- 风险扫描（数值膨胀/单一最优解/边缘情况）

Layer 2 发现问题 ≥1 项 → **必须修复**。

## 注意事项

1. **本 Skill 的输出是 GDD，不是最终游戏** — GDD 中的 `[待补充]` 标记是正常的，本节明确标注了哪些需要后续填充
2. **数值体系应在编码前完成最小可行设计** — GDD 的 §4 不应长期留空
3. **GDD 需要维护** — 随开发推进修改 GDD 是正常流程，不是"当初没写对"
4. **与 GDD 分析工作流配合使用** — 生成后建议运行 GDD 分析（`工作流/GDD分析.json`）进行第三方视角的验证
5. **如果没有框架设计怎么办** — 本 Skill 要求先有框架设计。如果还没有，先运行 `工作流/框架设计.json`
