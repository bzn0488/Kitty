# Skill 编写规范

> 本技能指导你如何在本工作区中创建、修改和审查 `.skill.md` 文件。Skill 是领域知识和方法论的沉淀，被 Agent 和 Prompt 按需调用。
>
> **重要**：设计新 prompt 应使用 `prompt设计` 相关技能，设计新工作流应使用 `设计工作流.prompt.md`（含引擎强制步骤）。本技能专注于 **Skill 文件本身**的编写规范。
>
> 另请参阅 `工作流与prompt设计通用经验.skill.md`，它包含了跨 workflow/prompt/skill 的通用原则。

## 🪶 快速启动（建一个新 skill 的 8 步）

按顺序执行，详细内容见对应章节：

```
1. 检查 scope 冲突     → 确认 _registry.json 中没有覆盖相同主题的已有 skill
2. 确定 layer           → domain(专业) / project(项目) / tool(工具)，放入对应目录
3. 精读知识库文章       → 用§5.1 蒸馏三步法提取内容
4. 写 .skill.md         → 按 template.skill.md，含 🪶 鸟瞰摘要
5. 写 .memory.md        → 记录初始化
6. 注册 _registry.json  → 加 layer/estimated_tokens，更新路径
7. 更新能力清单         → python _collect_capabilities.py
8. 验证                 → 检查能力清单 + 路径有效 + triggers 实测
```

> ⚠️ **容易遗漏**：
> - **Scope 冲突**：开始前检查现有 skill 是否已覆盖同一主题（避免重复建设）
> - **路径验证**：写完后验证 skill 中引用的所有文件路径是否有效（`技能/专业/xxx.md`）
> - **Triggers 实测**：注册后模拟一段对话，确认 triggers 能正确触发（而非"应该能触发"）
> - **子项目同步**：Junction 自动同步文件，但 `_registry.json` 需确保子项目也能访问

### 快速启动第 1 步详解：Scope 冲突检查

在写 skill 之前，检查是否有已有 skill 覆盖了相同或相近的主题：

1. **读取 `_registry.json`** — 遍历 `skills` 数组，查看每个技能的 `name`、`description` 和 `triggers`
2. **关键词交叉比对** — 用你新技能的预期主题关键词搜索已有技能的名字和 triggers
   - 例如你要写"心流体验"，搜索已有 skill 中是否有包含"心流"、"体验设计"、"Flow" 的
3. **重叠程度判断**：

| 程度     | 特征                                | 行动                                     |
| -------- | ----------------------------------- | ---------------------------------------- |
| 完全重叠 | 已有 skill 覆盖了完全相同的知识领域 | 不新建，改为改进现有 skill               |
| 部分重叠 | 新技能聚焦一个已有 skill 的子领域   | 新建，并在 §四 中注明与已有 skill 的边界 |
| 相互补充 | 相同领域的不同视角                  | 新建，并在两 skill 的参考中互相引用      |
| 无重叠   | ✔ 可以新建                          | —                                        |

4. **记录比对结果** — 在 `.memory.md` 的初始化记录中注明"与 XXX 无冲突"或"与 XXX 边界划分"

## 核心原则

### 原则 0：Skill 的读者是 Agent

`.skill.md` 的消费者是 AI agent，不是人。所有内容结构、措辞、格式都应围绕「如何让 agent 高效提取信息」来设计。agent 读取一个 skill 的完整路径是：

```
用户说话 → triggers 命中 → 加载 skill → 读 🪶 鸟瞰（去留决策）→
读核心原则（决策边界）→ 读内容主体 + §五（提取方法论）→
读自检清单（验证）→ 读 §四（边界确认）
```

这对 skill 写作者意味着：

| 如果你在...            | 那么...                                                                      |
| ---------------------- | ---------------------------------------------------------------------------- |
| 写 🪶 鸟瞰              | 这是 agent 的第一个去留决策点——写不好后面的内容再精彩也可能被跳过            |
| 在段落和表格之间做选择 | 优先用表格/列表——信息密度更高，agent 按行/列提取对比维度的效率远高于解析段落 |
| 写方案建议/设计决策    | 用"如果目标是 A → 用 X；如果目标是 B → 用 Y"——不说"可以考虑 X 或 Y"          |

> **来源**：COLLEAGUE.SKILL (arXiv:2605.31264) 的 progressive disclosure 设计——"agents see skill metadata first and load detailed instructions only when the skill is invoked"

**自检**：把 skill 文件贴在面前，模拟 agent 逐段读取——每读完一段，你是否清楚"下一段应该读什么"？如果某一段 agent 读完不知道该去哪，说明结构需要调整。

### 原则 1：Skill 是"工具箱"，不是"操作手册"

Skill 和 Prompt 的核心区别：

| 维度     | Skill (.skill.md)              | Prompt (.prompt.md)      |
| -------- | ------------------------------ | ------------------------ |
| 本质     | 领域知识和方法论沉淀           | 面向 AI 的详细操作指令   |
| 触发方式 | 自动加载（triggers 匹配）      | 手动调用（/prompt）      |
| 内容     | 原则、分类、自检清单、经验教训 | 步骤、输出格式、约束条件 |
| 强制力   | 无（参考性知识）               | 有（步骤依赖链）         |
| 类比     | 专家的"工具箱"                 | 专家的"操作手册"         |

**检查**：你写的内容是"知识/原则/经验"还是"步骤/指令"？如果是后者，可能更适合做成 Prompt。

### 原则 1.2：游戏设计 skill 的范围约束

游戏设计相关的 skill 应**只覆盖策划/设计工作**，不覆盖音效、美术、程序实现等非策划领域，除非该 skill 的主题明确需要跨界（如"UI/UX 设计"）。

| 领域                         | 属于设计 skill？ | 说明                                    |
| ---------------------------- | :--------------: | --------------------------------------- |
| 系统设计、框架设计、数值平衡 |      ✅ 核心      | 纯策划工作                              |
| 卡牌/武器/事件设计           |      ✅ 核心      | 策划核心产出                            |
| 反馈系统、驱动力、心流       |      ✅ 核心      | 设计理论支撑                            |
| 游戏节奏、体验设计           |      ✅ 核心      | 策划工作                                |
| **音效设计、配乐、音频反馈** |        ❌         | 除非 skill 主题是"音频设计"，否则不考虑 |
| **美术风格、UI 视觉设计**    |        ❌         | 除非 skill 主题是"UI/UX"，否则不考虑    |
| **程序实现、性能优化**       |        ❌         | 技术工作，不由设计 skill 覆盖           |

**原因**：技能加载消耗 token 预算。如果每个设计 skill 都附带音效/美术建议，token 利用率会大幅下降，且 Agent 可能给出一堆"此处应有音效"之类的外行建议。

**自检**：创建设计 skill 时，检查是否包含了非策划内容。如果有，单独拆成独立的"音效设计"、"UI 设计"等 skill，或者在 §四 中注明"本 skill 不覆盖 XXX"。

### 🪺 原则 1.5：创建是识别的机会

在写一个 skill 的过程中，**几乎一定会**发现其他值得创建的 skill 候选。这时必须：

1. **立即记录** — 打开 `文档/待建Skill清单.md`，追加新条目（主题 + 一句话理由 + 参考来源）
2. **关联说明** — 在本次创建的 `.memory.md` 初始化记录中注明"创建 X 时发现 Y 值得建"
3. **优先级标记** — 参考已有清单，给新条目标 P1/P2/P3

> **为什么**：因为"正巧在读反馈相关的知识库文章，顺手就能判断哪些主题适合独立成 skill"。等到以后专门做规划时，上下文已经冷了，遗漏概率大。
>
> **自检**：如果 `.memory.md` 中没有"创建时发现的候选 skill"条目，说明这个步骤被跳过了。

### 原则 2：Frontmatter 是技能的身份证

Frontmatter 是 YAML 格式的元数据块，决定了技能何时被触发、能做什么。

```
---
id: skill_writing_guide           # ✅ 全局唯一，小写+下划线
name: Skill 编写规范               # ✅ 人类可读的中文名称
description: "..."                 # ✅ 一段话说明技能用途和范围
triggers:                          # ✅ Agent 自动加载的关键词
  - 写skill
  - 技能设计
tools:                             # ✅ 技能可用工具的白名单
  - read_file
  - create_file
output_schema:                     # ✅ 期望的输出结构（可选）
  skill_id: "string"
---
```

**字段详细规则**：

| 字段               | 必填  | 规则                                                                                                                        |
| ------------------ | :---: | --------------------------------------------------------------------------------------------------------------------------- |
| `id`               |   ✅   | 小写字母 + 下划线，全局唯一（例：`framework_design`）                                                                       |
| `name`             |   ✅   | **中文名称**，一眼能看出技能用途。公认的英文术语缩写（GDD、UI、AI、RPG 等）可以保留。文件名也使用中文，与 `name` 保持一致。 |
| `description`      |   ✅   | 一段话说明——这是 Agent 判断是否加载的关键文本，必须包含"当 XXX 时自动加载"的暗示                                            |
| `version`          |   ✅   | 语义化版本号 `X.Y.Z`——X 大改（结构变化）、Y 中改（内容增删）、Z 小改（勘误）                                                |
| `layer`            |   ✅   | `domain`（专业）/ `project`（项目）/ `tool`（工具）——见 `文档/Skill框架重构方案.md` §2.1                                    |
| `estimated_tokens` |   🟡   | 基于文件大小的预估 token 数（用于编排器预算管理）                                                                           |
| `triggers`         |   ✅   | 触发关键词列表——见下方"Triggers 设计"                                                                                       |
| `tools`            |   ✅   | 工具白名单——见下方"Tools 选择"                                                                                              |
| `output_schema`    |   🟡   | 期望的输出结构——如果技能有结构化产出，定义字段类型                                                                          |

### 原则 3：正文结构遵循「鸟瞰→能力→行为边界→回流」

一个完整的 Skill 文件正文应按以下顺序组织：

```
# 标题（一句话定位）

> 摘要/背景说明（可选，引用关联文件）
> **版本**: X.Y.Z

---

## 🪶 鸟瞰摘要（必须，除非总 token < 500）
一段速查/概览，让 Agent 一眼判断是否需要深入。这是 frontmatter 之后 agent 的第一个"去留决策点"——读完后 agent 决定是否继续深入后续内容。

**必须满足**：
- 表格或列表格式（不能是段落，段落对 agent 的信息提取效率最低）
- 每行 ≤ 15 字
- 不包含原理说明（那是正文的事）

通常是：
- 一张汇总表（行=技法/场景，列=一句话+最佳场景）
- 或 3-5 个核心要点（覆盖问题域、方法分类、适用条件）

**可省略条件**：技能总 token < 500（此时 agent 直接读全文成本极低，鸟瞰收益可忽略）

> 来源：Corpus2Skill (arXiv:2604.14572) — "Agent 从鸟瞰图→逐级摘要→原文逐层导航"。原则 0 解释了为什么鸟瞰是 agent 的"去留决策点"

---

## 核心原则
2-5 条设计/方法论原则，每条含检查和示例

## 内容主体（能力轨道）
包括不限于：
- 分类标准（如框架类型分类、正反馈技法分类）
- 设计维度和对比表格
- **决策启发式** — 不只说"怎么做"，还要说"什么情况下选这个方案"
  - 例如：「如果目标是让玩家感到成长→用叠层技法；如果目标是制造惊喜→用投篮效应」
- 可操作的方法/步骤

## 快速自检清单
创建或修改后可逐条验证的 check 列表

---

## §四 行为边界（原"经验教训"，扩展）
> 告诉 Agent「什么情况下不适用」和「已经踩过的坑」。

### 4.x 反模式
描述做错了什么、后果、怎么避免。

### 4.x 不适用场景
描述这个方案/原则在什么条件下不适用。
例如：「可变比率强化在单次博弈（非长期游戏中效果有限）」

> 来源：COLLEAGUE.SKILL (arXiv:2605.31264) — 行为边界轨道

## §五 设计方法论（能力轨道）
> 正向引导——告诉 Agent「应该怎么做」。

### 5.x 方法/模型/公式
描述设计方法、使用步骤、判断条件。

---

## 参考
- 关联文件路径
- 知识库文档
- 外部资源
```

**§四 和 §五 是可选但推荐的章节**，前者记录"不要做什么"（负面约束、自检项），后者记录"应该怎么做"（正面方法论）。参见 `工作流与prompt设计通用经验.skill.md` 的实际案例。

### 使用后回流

每次使用本技能完成审查或创建后，**立即**在本技能的 `.memory.md` 中追加一条记录：
- 本次审查/创建的目标文件
- 发现了什么问题（边界案例、未覆盖情况）
- 本技能自身是否有改进空间

> ⚠️ **自指约束**：本技能的使用者也需遵守此条。本次不写就违背了规范本身。

## Triggers 设计

Triggers 决定了技能何时被自动加载到 Agent 的上下文中。

### 选择规则

1. **覆盖常用说法** — 用户可能用不同的词表达同一需求
   - 例如：`["写skill", "技能设计", "创建技能", "skill编写", "怎样写skill"]`
2. **避免过于宽泛** — 一个词的 trigger 可能误触发
   - ❌ `["游戏"]` → 太宽泛，任何游戏相关对话都会触发
   - ✅ `["框架设计", "系统链接", "相对最优解"]` → 精准匹配
3. **以中文为主，术语缩写可保留英文** — `name` 和文件名用中文方便查找，triggers 可中英混合覆盖用户的不同说法
   - `name`: `"正反馈技法"` ✅ / `"GDD生成"` ✅（GDD 是术语缩写）
   - `name`: `"Feedback System Design"` ❌（在中文工作区中不易查找）
   - 文件名：`正反馈技法-系统框架.skill.md` ✅ / `GDD生成.skill.md` ✅
4. **不要包含常见停用词** — `的`、`了`、`是`、`什么` 等不适合
5. **数量 3-10 个** — 太少可能漏触发，太多可能误触发

### 检查方法

自检 triggers 质量：
- 取任一 trigger，问"用户说这个词时，是否确定需要这个技能？"
- 如果答案是否定的 → 这个词太宽泛，需要删或换
- 如果某个合理的触发场景没有被任何 trigger 覆盖 → 需要加

### 测试方法

注册后建议实测 triggers 是否生效：

1. **准备一组测试话术** — 选 3-5 个预期触发场景的自然说话方式
   - 例如对"框架设计"技能："帮我设计一个游戏框架"、"这个系统的结构怎么做"
2. **找一个未打开的聊天窗口** — 发送测试话术
3. **验证加载** — 检查技能是否出现在 Agent 的上下文加载列表中
4. **边界测试** — 发一个预期**不应该**触发的话术（如"今天天气不错"），确认不误触发
5. **修正** — 如果漏触发则加 trigger，如果误触发则删/改 trigger

## Tools 选择

Tools 白名单限制了技能执行时可用的工具。选择原则：

1. **够用但不冗余** — 只加这个技能确实会用到的工具
   - 不需要 `run_in_terminal` 的纯知识技能就不要加
2. **常用工具参考**：

| 工具                     | 适用场景                  |
| ------------------------ | ------------------------- |
| `read_file`              | 读取现有文件（几乎必选）  |
| `file_search`            | 搜索文件路径              |
| `grep_search`            | 搜索文件内容              |
| `semantic_search`        | 语义搜索（知识/代码查询） |
| `create_file`            | 创建新文件                |
| `replace_string_in_file` | 修改已有文件              |
| `run_in_terminal`        | 执行命令行（运行脚本等）  |
| `memory`                 | 读写记忆文件              |

3. **安全考虑**：技能不会自动获得工具——没有在 tools 中列出的工具，技能无法调用。所以 review 时检查 tools 是否足够，是最常见的遗漏。

## 注册流程

创建/修改技能后，按此流程完成注册：

```
1. 创建/修改 .skill.md  ← 你现在在这里
       ↓
2. 创建配套 .memory.md  ← 记录使用经验
       ↓
3. 注册到 _registry.json ← 添加技能条目
       ↓
4. 运行能力清单更新     ← python _collect_capabilities.py
       ↓
5. 验证生效             ← 检查能力清单是否包含新技能
```

### 步骤详解

**步骤 2 — 创建 .memory.md**
- 文件命名：`<技能文件名>.memory.md`（与 .skill.md 同名同目录）
- 内容结构：见 `工作流与prompt设计通用经验.memory.md` 的格式
- 即使首次创建也填一条初始化记录

**步骤 3 — 注册 _registry.json**
- 在 `_shared/技能/_registry.json` 的 `skills` 数组中追加
- 每个条目包含：`id`, `name`, `description`, `triggers`, `prompt`, `tools`, `output_schema`
- **区分 `description` 和 `prompt` 的受众**：
  - `description` → 给人看的，在技能目录中展示，说明技能用途和范围
  - `prompt` → 给 agent 看的预筛选文本，agent 读这段来决定是否加载完整 skill 文件
- `prompt` 字段应该**一句话说清**：什么场景用 + 包含什么 + 什么场景**不要**用

完整条目示例（加在数组末尾，注意前一项后的逗号）：
```json
{
  "id": "positive_feedback",
  "name": "正反馈技法",
  "description": "设计游戏中的正反馈与负反馈循环、信号强化体系、双轨收益框架",
  "triggers": ["正反馈", "反馈循环", "强化体系", "双轨收益", "反馈设计"],
  "prompt": "当需要设计游戏中的反馈系统时使用。提供正/负反馈循环的设计原则、强化程式分类表、以及自检清单。",
  "tools": ["read_file", "file_search", "grep_search"],
  "output_schema": {
    "feedback_type": "string",
    "reinforcement_schedule": "string",
    "dual_track": "object",
    "checklist_passed": "boolean"
  },
  "version": "1.0.0",
  "layer": "domain",
  "estimated_tokens": 8000
}
```
注意新增字段 `version`、`layer`、`estimated_tokens` 与 skill 文件 frontmatter 保持一致。

**步骤 4 — 运行能力清单**
```bash
python _共享资源/脚本/_collect_capabilities.py
```
- 自动扫描所有 `.skill.md` / `.prompt.md` / `.agent.md` / 工作流 `.json` 并更新 `_共享资源/能力清单.md`
- 如果忘记运行，能力清单会与实际情况不一致

**步骤 5 — 验证**
- 检查能力清单中是否出现了新技能
- 检查 `_registry.json` 的语法是否有误（JSON 必须合法）

## 内容组织技巧

### 1. 对比表格法

用表格对比两个容易混淆的概念，是 Skill 中最有效的知识传递方式：

```
| 维度 | ❌ 错误概念 | ✅ 正确概念 |
| ---- | ---------- | ---------- |
| 范围 | ...        | ...        |
| 目标 | ...        | ...        |
```

### 2. 自检清单法

每个原则/概念后附一个自检清单，让其可操作：

```
**自检清单**：
- [ ] 条件 1
- [ ] 条件 2
```

### 3. 正反例法

```
| 错误做法   | 正确做法   |
| ---------- | ---------- |
| "不要做 X" | "应该做 Y" |
```

### 4. 引用关联法

在技能中引用其他技能、prompt 或知识库文档，形成知识网络：

```
参见 `工作流与prompt设计通用经验.skill.md` 的原则 3
参见 `_共享资源/知识库/理论知识/...`
```

### 5. 内容格式决策表

选择用什么格式呈现代码知识，取决于 agent 需要做什么：

| 信息类型  | 推荐格式                | 原因（agent 视角）                  |
| --------- | ----------------------- | ----------------------------------- |
| 分类/对比 | 表格                    | agent 按行/列提取对比维度的效率最高 |
| 判断条件  | "如果 X → 用 Y" 列表    | 条件分支最容易被 agent 逐条匹配     |
| 步骤流程  | 编号列表                | agent 按序号顺序执行最不容易跳步    |
| 概念定义  | 一句精简定义 + 示例     | agent 需要精确理解边界后套用        |
| 经验教训  | 症状/根因/修复三段式    | agent 先判断是否匹配症状，再读修复  |
| 复杂推理  | 段落（控制在 3-5 句内） | 仅在需要建立上下文依赖时使用        |

**检查**：如果一段内容超过 50 字且不是表格/列表，问自己"这段能改成表格或列表吗？"——如果可以，改了 agent 读得更快。

## 快速自检清单

创建或修改完技能后，逐条检查：

```
[ ] Frontmatter 完整（id/name/description/version/layer/triggers/tools）
[ ] id 是唯一的小写+下划线格式
[ ] version 已更新（创建为 1.0.0，修改后适当递增）
[ ] layer 与文件所在目录一致（domain→专业/，project→项目/，tool→工具/）
[ ] estimated_tokens 已估算（基于文件大小）
[ ] description 清晰说明技能用途和触发场景
[ ] triggers 覆盖主要使用场景，无过于宽泛的词
[ ] tools 足够执行本技能需要的工作，无多余工具
[ ] 正文包含 🪶 鸟瞰摘要（推荐）或至少一个核心概念定义
[ ] 有自检清单或可操作的方法论
[ ] 配套的 .memory.md 已创建（即使只有初始化记录）
[ ] _registry.json 已注册（含 layer/estimated_tokens/新路径）
[ ] 能力清单已更新（运行 _collect_capabilities.py）
[ ] §四 包含行为边界（不适用场景）或经验教训
[ ] 路径引用使用新子目录格式（技能/专业/xxx.skill.md）
[ ] §五 篇幅 > §四（正向引导为主，负面约束为辅）
[ ] 章节顺序符合 top-down：鸟瞰→原则→主体→自检→§四→§五
[ ] 每章有明确的防错目标（可回答"这章防止什么"）
[ ] Agent 可读性 — 读完 🪶 鸟瞰能在 3 秒内判断是否需要此 skill
[ ] Agent 可读性 — 决策启发式全部以"如果 X → 用 Y"形式呈现
[ ] Agent 可读性 — 表格/列表使用与段落使用的比例符合原则 0 的"表格优先"建议
[ ] 范围约束 — 设计 skill 中没有混入音效/美术/程序等非策划内容（除非 skill 主题明确需要）
[ ] Frontmatter 使用 `---` 分隔符（非 ` ```yaml `）
[ ] 正向引导优先——正文开头以"是什么/怎么做"而非"不是X/不要X"开头
```

## 四、经验教训

> 已验证的反模式和风险点，来自现有技能库的实战回流。

### 4.1 技能与 prompt 功能混淆

**症状**：技能文件里写了详细的步骤指令（像 prompt），或者 prompt 文件里混入了大段知识（像 skill）。

**根因**：不清楚 skill 和 prompt 的分工。Skill 是"工具箱"（知识/原则/经验），Prompt 是"操作手册"（步骤/指令/约束）。

**自检**：打开技能文件，如果超过 50% 的内容是"先做 A，再做 B，然后输出 C"的步骤式指令，说明这个内容更适合做成 prompt。

### 4.2 triggers 设计失当

**症状**：用户说了相关话题但技能未被加载（漏触发），或者无关话题加载了技能（误触发）。

**常见模式**：
- trigger 太少 → 漏触发，技能白写了
- trigger 太宽泛 → 误触发，污染上下文
- trigger 只有英文 → 中文用户说话时无法触发

**自检**：检查 triggers 是否符合"取任一 trigger，用户说这个词时，是否确定需要这个技能"。

### 4.3 缺少不适用场景说明

**症状**：技能只说了"怎么做"，没说"什么情况下不该这么做"。Agent 错误地套用了不合适的方案。

**根因**：§四 只记录了反模式（做错了），没有记录适用边界（什么情况下不适用）。

**自检**：检查 §四 是否包含"不适用场景"条目。例如可变比率强化在单次博弈游戏中效果有限。

### 4.4 忘记注册和能力清单更新

**症状**：技能文件写好了，但 `_registry.json` 没有对应条目，能力清单也看不到。

**根因**：写完后只记得保存文件，忘记后续步骤。

**修复**：注册流程中的步骤 3 和 4 不可跳过。建议每次创建技能后立即按流程执行。

### 4.5 Frontmatter 格式——用 ```yaml 代替标准 ---

**症状**：技能文件开头使用 ` ```yaml ... ``` ` 代码块而非标准的 `---` 分隔符。

**根因**：不清楚 Agent 的 frontmatter 解析器只识别 `---` 分隔符，不识别代码块。` ```yaml ` 会被当作普通 markdown 内容。

**严重程度**：⚫ **阻断性**——triggers/tools/description 全部不会被解析，Agent 依赖 `_registry.json` 兜底才能加载。

**修复**：始终使用 `---` 作为 frontmatter 分隔符。

> 实战数据：本次审计发现 7 个技能中有 5 个使用了 ` ```yaml ` 格式（占 71%）。之前它们能工作完全依赖 `_registry.json` 的独立配置。

### 4.6 正向引导优先原则被忽视

**症状**：skill 正文以"不是 X"、"不要做 X"、"避免 X"的否定句开头。

**根因**：想用"澄清误解"来建立概念边界，但效果是 agent 先读到四条禁令而非"怎么做"。

**修正**：否定内容放到 §四（行为边界 > 反模式），正文开头用正向定义。例如：

```
❌ 权力下放不是功能/不是恩赐/不是变简单/不是放弃控制
✅ 权力下放是开发者将控制权有策略地让渡给玩家
```

**实战教训**：本次创建 `权力下放设计.skill.md` 时也犯了同样错误，被用户指出后才修正。

### 4.7 哲学型 skill 的"委托"模式

**症状**：技能内容膨胀，试图在一个 skill 中装下所有相关内容。

**根因**：skill 粒度过粗，没有区分"哲学层"和"技术实现层"。

**新模式——哲学委托**：某些 skill 更适合作为"哲学定位 + 各层引用"，将具体方法论委托给专用 skill。

```
权力下放设计.skill.md（哲学）
  ├→ 框架层：委托给 框架设计.skill.md
  ├→ 系统层：委托给 心流与体验设计.skill.md
  └→ 具体层：委托给 正反馈技法-具体设计.skill.md
```

**适用判断**：如果 skill 的核心理念适用于多个层面，且每层都有对应的专用 skill，考虑用"哲学委托"模式而非膨胀内容。

## 五、知识提取方法论

> 正向引导——从知识库文章到 Skill 的高密度蒸馏方法。

### 5.1 知识蒸馏三步法

**适用场景**：从知识库文章（如 `理论知识/46-正反馈与负反馈循环.md`）创建新的专业 skill。

**第一步：精读与分类**

通读知识库文章，将内容分为三类：

| 标记  | 类型                     | 处理方式             | 示例                         |
| :---: | ------------------------ | -------------------- | ---------------------------- |
|   🔵   | **原理/定义/理论推导**   | **留在知识库**       | 正反馈的生物学机制           |
|   🟢   | **可操作原则/方法/分类** | **提炼到 skill**     | 叠层技法、反差技法、自检清单 |
|   🟡   | **案例/示例**            | **精简后放入 skill** | 游戏案例可压缩为 1-2 句话    |

**判断标准**：如果一段文字去掉后不影响 Agent 执行任务 → 留在知识库。如果去掉后 Agent 不知道"该怎么做" → 放入 skill。

**第二步：按 skill 结构组织**

将 🟢 内容填入 skill 框架：

```
🪶 鸟瞰摘要     ← 一张汇总表，覆盖核心知识点
核心原则        ← 从文章中提取 2-5 条通用原则
内容主体        ← 分类/维度/对比表格 + 决策启发式
自检清单        ← 每个原则至少对应一条自检项
§四 行为边界    ← 反模式 + 不适用场景
§五 方法论      ← 方法步骤/模型公式
参考            ← 标注来源文章路径
```

**第三步：标注交叉引用**

skill 中的每个核心知识点标注来源 KB 文章：

```markdown
| **叠层** | [来源:46-正反馈与负反馈循环.md] 小奖励累积触发大奖励 |
```

### 5.2 内容密度原则

Skill 是知识库的**高密度蒸馏版**：

```
源文章 (~5000 字：理论 + 案例 + 推导)
     ↓ 蒸馏
Skill (~800-2000 字：定义 + 原则 + 分类 + 自检 + 反模式 + 精简案例)
```

**密度检查**：读完 skill 后如果 Agent 还需要打开知识库文章才能理解 → skill 太薄了。如果 skill 里出现了三段以上的连续 prose 段落 → skill 太厚了。

### 5.3 决策启发式写法

不只说"怎么做"，还要说"什么情况下选这个方案"：

```
❌ 叠层技法：小奖励累积到阈值触发大奖励（只说了是什么）

✅ 叠层技法：小奖励累积到阈值触发大奖励
   适用场景：玩家需要持续的小正反馈维持参与度
   不适用：单次体验（如叙事游戏）
   竞争方案：如果目标是惊喜而非成长 → 投篮效应更合适
```

### 5.4 🪶 鸟瞰摘要写法

skill 顶部的速查段，让 Agent 一眼判断是否需要深入：

```
## 🪶 正反馈技法速查

| 技法     | 一句话                   | 最佳场景         |
| -------- | ------------------------ | ---------------- |
| 叠层     | 小奖励→阈值→大奖励       | 数值成长型       |
| 投篮效应 | 低成本高频+小概率高回报  | 操作型/风险型    |
| 反差     | 强化态与基准态反差足够大 | 有状态切换的游戏 |
```

**规则**：
- 必须是**表格或列表**，不能是段落
- 每行不超过 15 字
- 不包含原理说明（那是正文的事）

### 5.5 最小可行法：先发布再完善

首次创建技能不需要完美：

```
V1: 标题 + 鸟瞰摘要 + 核心定义 + 自检清单  ← 发布
V2: + 核心原则扩展                       ← 使用后补充
V3: + §四 行为边界                       ← 遇到反模式后添加
V4: + §五 决策启发式                     ← 形成判断后添加
V5: + 参考链接                           ← 关联其他技能/文档
```

### 5.6 每章为什么：skill 各章节的防错目标

skill 的每个章节都有其存在的理由——防止特定的设计失误：

| 章节                      | 防错目标                                       | 如果缺失会怎样                              |
| :------------------------ | :--------------------------------------------- | :------------------------------------------ |
| 🪶 **鸟瞰摘要**            | 让 Agent 快速判断这个 skill 是否适用于当前任务 | Agent 读完整个 skill 才发现不适用，浪费时间 |
| **核心原则**              | 提供设计决策的底层逻辑                         | Agent 只会套用模板，不理解"为什么这么做"    |
| **内容主体（分类/维度）** | 提供选择方案的判断依据                         | Agent 用错分类，选错方案                    |
| **决策启发式**            | 在多个可行方案中做正确选择                     | Agent 在错误场景下用了正确的方法            |
| **自检清单**              | 方案完成后提供可验证的质量门禁                 | 设计缺陷在展示给用户后才被发现              |
| **§四 行为边界**          | 阻止在不适用的场景中使用本技能                 | Agent 把场景 A 适用的方法错误套用到场景 B   |
| **§五 方法论**            | 提供可操作的具体步骤                           | Agent 知道"该怎么做"但不知"具体怎么做"      |
| **参考**                  | 遇到 skill 未覆盖的深层问题时提供退路          | Agent 在知识边界上卡住，无路可退            |

**来源**：`_共享资源/文档/设计工作流元经验.md` 第 6 条"每步为什么"

### 5.7 Top-down 内容组织

skill 各章节的排列顺序不是随意的，而是 top-down 思维驱动的：

```
Agent 读取 skill 的顺序应该是：
1. 这是什么？     → 🪶 鸟瞰摘要      ← 定位
2. 为什么这么做？   → 核心原则        ← 原理
3. 具体怎么做？     → 内容主体 + §五  ← 执行
4. 选哪个方案？     → 决策启发式      ← 判断
5. 做完了怎么检查？ → 自检清单        ← 验证
6. 什么情况不适合？ → §四 行为边界    ← 边界
```

每一步依赖上一步的理解——先知道"是什么"，再理解"为什么"，然后才是"怎么做"。如果 Agent 跳过前面的步骤直接读 §五（方法论），可能会因为不理解上下文而误用。

**来源**：`_共享资源/文档/设计工作流元经验.md` 第 5 条"Top-down 比 Bottom-up 好"

### 5.8 正向引导优先

skill 的主体内容应以**正向方法论**为主（"应该怎么做"），§四 行为边界是辅助性的。

**比例建议**：

```
§五 方法论 + 内容主体    ≈ 70-80%  ← 主要篇幅
§四 行为边界             ≈ 10-15%  ← 辅助
自检清单                  ≈ 5-10%  ← 回顾性验证
```

**原因**（来自元经验第 4 条）：
- Agent 加载 skill 时最需要的是"告诉我该怎么做"，不是"别做什么"
- 自检清单和边界条件在方案**设计完成后**回顾时才用得上
- 过多的负面约束会抑制创造力——Agent 可能避开字面禁止的内容但设计出等价的替代方案

**检查**：如果 §四 的篇幅超过了 §五，说明这个 skill 的反模式太多——要么是知识提炼不够（误把反模式当核心内容），要么是这个主题本身风险极高（此时反模式多是合理的）。

**来源**：`_共享资源/文档/设计工作流元经验.md` 第 4 条"正向引导优于负面约束"
