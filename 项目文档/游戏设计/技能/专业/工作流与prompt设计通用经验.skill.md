# 工作流与 Prompt 设计通用经验

> 本文件是对 `_共享资源/文档/设计工作流元经验.md` 的提炼摘要，适用于工作流定义（JSON）、Prompt 文件（.prompt.md）和 Skill 文件（.skill.md）的设计与修改。
> **重要**：设计新工作流时应使用 `设计工作流.prompt.md`（含引擎强制步骤），本技能仅作为轻量参考和自检。

## 核心原则

### 原则 1：每步必须有不可跳过的中间产物

| 错误做法                     | 正确做法                                                          |
| ---------------------------- | ----------------------------------------------------------------- |
| "分析问题"→ 没有产出物格式   | "分析问题"→ 要求输出 `findings: array` + `improvements: array`    |
| "概念发散"→ Agent 一句话带过 | "概念发散"→ 要求至少 3 个概念，每个附核心体验 + 参考来源          |
| "请考虑..."→ 模糊要求        | "产出：六维度驱动力分析表（选择情境/内驱/外驱/排斥力/阈值/校对）" |

**检查**：每个步骤是否定义了具体的、结构化的输出格式（表/字段/分类标准）？

### 原则 2：设计预览确认点

任何涉及"产出设计方案"的流程，必须在设计完成后、写入文件前，设置一个**预览→确认→退回**环节。

**检查**：
- 用户能否在设计方案写死之前看到它？
- 预览格式是否明确指定（不是"展示一下"而是具体的模板格式）？
- 用户确认后才执行下一步（`human_confirm: true`）？

### 原则 3：正向引导，不要禁令

| 错误做法         | 正确做法                                             |
| ---------------- | ---------------------------------------------------- |
| "不要设计诅咒卡" | "每个选项应体现资源转换——消耗什么得到什么"           |
| "不能有功能重叠" | "每张卡应有差异化定位，从不同角度服务于同一设计目的" |

**检查**：Prompt 中是否主要使用"应该怎么做"（正向）而非"不准做X"（负面）？负面约束应放入自检清单而非步骤指引中。

### 原则 4：步骤顺序反映思维流程

设计流程的步骤顺序应反映 top-down 思维过程，而非自下而上的拼装。

```
正确：场景分析 → 设计目的 → 概念发散 → 行为预检 → 具体设计 → 自检 → 预览
错误：具体设计 → 找理由 → 补场景
```

**检查**：步骤序列是否体现"先理解处境→再探索可能→后收敛方案"的自然思维过程？

### 原则 5：知识回流必须写死

经验提炼不能依赖 Agent 的"自觉性"，必须是工作流的固定步骤。

**检查**：
- 工作流的最后是否有一个步骤要求追加 `.memory.md` 或 `决策日志`？
- 如果修改了 `_共享资源/` 下的文件，是否触发了能力清单更新？

## 快速自检清单

每次设计/修改完工作流或 Prompt 后，逐条检查：

```
[ ] 每步有不可跳过的结构化中间产物（不只是"分析一下"）
[ ] 有预览确认点（human_confirm 或明确的展示 + 用户确认流程）
[ ] 指引以正向引导为主（"该怎么做">"不准做X"）
[ ] 步骤顺序是先发散后收敛（不出现先定方案再发散的倒置）
[ ] 知识回流已写死（.memory.md / 决策日志）
[ ] 引擎路径正确（通过 `工作流/` 可访问——模板根目录和子项目均为 Junction）
[ ] 如果新增了工作流路由 → 步骤 ID 与 prompt 中的步骤一一对应
[ ] 如果新增了 prompt/skill → 运行 `_collect_capabilities.py` 更新能力清单
```

## 四、经验教训

> 已验证的反模式和风险点，来自实战回流。

### 4.1 强化程式单一依赖

可变比率强化虽然效果最佳、行为最持久，但过度依赖单一强化程式会导致：
- 玩家对固定节奏产生耐受，正反馈衰减加速
- 当强化程式变化时，玩家行为出现剧烈波动

**自检**：检查设计中是否存在过度依赖单一强化程式的情况？是否混合了多种程式（固定比率兜底 + 可变比率惊喜）？

### 4.2 心理力学 vs MDA 混淆

MDA 框架常被误用作设计起点。心理力学模型回答「玩家为什么改变行为」，MDA 回答「设计是否产生了预期体验」。两者用途不同。

**自检**：设计过程中是否先用心理力学框架思考行为驱动，完成后才用 MDA 验证？

### 4.3 不适用场景说明

本技能的原则和自检清单**不适用于**以下场景：
- **纯技术实现文档**（API 设计、数据库 Schema）——需使用各自领域的设计规范
- **已有成熟方法论覆盖的领域**（如 Scrum 流程、UI 设计系统）——不应与现有方法论冲突
- **单次执行的脚本任务**（数据清洗、文件转换）——没有多步骤需要设计
- **已经有专用 skill 覆盖的场景**（如正反馈技法、卡牌设计）——优先使用专用 skill 而非通用原则

**自检**：当前设计对象是否属于上述不适用场景之一？如果是，考虑使用更专用的方法论。

## 五、设计方法论

> 正向引导——告诉 Agent「应该怎么做」，来自专业知识回流。

### 5.1 心理力学模型（行为改变公式）

设计时评估玩家是否会按照预期改变行为，使用以下公式逐项分析：

```
行为改变倾向 = 过程吸引度 + 结果吸引度
             - 过程排斥度 - 结果排斥度
             - 当前行为惯性 - 改变成本
```

**用法**：
- 对每个核心设计决策，逐项评估六个因素的相对强弱
- 如果「惯性 + 改变成本」>「吸引度 - 排斥度」→ 玩家不会改变行为
- 要推动行为改变：要么提高吸引度，要么降低惯性/改变成本

**设计检查点**：
| 因素         | 检查问题                                           |
| :----------- | :------------------------------------------------- |
| 过程吸引度   | 行为过程中玩家是否感到愉悦？核心玩法是否足够有趣？ |
| 结果吸引度   | 行为结果的正反馈是否足够强？奖励是否值得期待？     |
| 过程排斥度   | 行为过程中是否有挫败感、无聊或疲劳？               |
| 结果排斥度   | 行为失败是否有过重的惩罚？损失是否导致放弃？       |
| 当前行为惯性 | 玩家是否有路径依赖？沉没成本是否让ta不愿切换？     |
| 改变成本     | 学习新行为的成本有多高？切换门槛是否合理？         |

### 5.2 心理力学是思考基点，MDA 是验证工具

| 框架         | 用途                                            | 使用时机       |
| :----------- | :---------------------------------------------- | :------------- |
| 心理力学模型 | 回答「玩家为什么改变/维持行为」——设计的思考基点 | **设计过程中** |
| MDA框架      | 回答「机制→行为→体验的链条是否成立」——验证工具  | **设计完成后** |

### 5.3 设计决策启发式

在不同场景下选择不同设计策略：

| 如果目标...                      | 则...                                                                    |
| -------------------------------- | ------------------------------------------------------------------------ |
| 让每步不落空                     | 给每步定义结构化输出（字段/表/分类标准）——不写"分析一下"之类的模糊要求   |
| 需要用户确认设计                 | 加 `human_confirm: true` 或预览→确认→退回环节                            |
| 想防止 Agent 踩已知的坑          | 在 §四 写反模式（症状→根因→修复），而不是在步骤里加"不要做X"禁令         |
| 内容偏负面约束（不要做X）        | 把禁令改为正向引导 + 自检清单——"不准设计诅咒卡"→"每个选项应体现资源转换" |
| 步骤顺序不确信                   | 按照场景分析→目的→概念→预检→设计→自检→预览 的 top-down 顺序              |
| 需要经验不被遗忘                 | 在工作流最后加一个强制步骤写 `.memory.md` 和决策日志                     |
| 心理力学公式计算显示改变成本过高 | 降低学习门槛或提高过程吸引度，而不是继续堆叠结果奖励                     |
